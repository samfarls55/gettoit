## What to build

A research spike — **no application code ships**. Sample the live Foursquare `tastes` field and produce a curated vibe-token allowlist that the tb-18 classifier consumes.

`tastes` is a crowd-sourced free-text tag cloud (the issue cites ~76% venue coverage) carrying genuine atmosphere tokens (`quiet`, `lively`, `cozy`, `intimate`, `crowded`, `trendy`, `romantic`, `good for groups`, `spacious`, …) buried in folksonomy noise (`trains`, `hummingbirds`, chef names, dish names).

Deliverable — a vault research note containing:

1. A token-frequency sample of the live `tastes` field across a representative venue pool, pulled via the places-proxy / `GET /places/search` with `tastes` named in the `fields` csv (`tastes` is already in the proxy's field list — no new API cost).
2. The curated **vibe-token allowlist** — a flat list of tokens, each tagged with one direction: `+1` (loud-leaning — lively, crowded, rowdy, …) or `-1` (quiet-leaning — cozy, intimate, quiet, …). No magnitude/weight column; the grill locked direction-only. Folksonomy noise and atmosphere-ambiguous tokens are excluded.
3. The observed `tastes` coverage rate over the sample (confirm or correct the ~76% figure against live data).

The allowlist's *consumption* contract is already fixed by tb-18 (see tb-18 §"Design — locked during triage"): the classifier sums the ±1 tags of a venue's matched tokens and nudges the category-archetype baseline by the **sign** of that sum, capped at ±1. This spike only produces the token set and tags — it does not touch classifier code.
