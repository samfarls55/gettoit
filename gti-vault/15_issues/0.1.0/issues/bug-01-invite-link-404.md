# bug-01 — Invite link 404

## Parent

[[../_index|0.1.0 backlog]] candidate #1. Source: [[../testflight-first-dogfood-2026-05-14|TestFlight dogfood 2026-05-14]].

## What's broken

On the first real-device install (2026-05-14 TestFlight), the invite link generated by the iOS app routes to a 404 instead of the intended web fallback or iOS deep-link target. Breaks the invite flow end-to-end — viral loop is dead until this resolves.

Investigate alongside [[bug-02-static-og-image-placeholder|bug-02]] — both point at AASA / universal-links plumbing from TB-00 + TB-02; closing one may close the other.

## Likely root-cause candidates

- AASA file (`web/public/.well-known/apple-app-site-association`) is served but the path matching does not include the live `/join/<roomId>` shape, so Universal Link handoff falls through to the browser, which loads a non-existent route.
- Web app's `/join/[roomId]/page.tsx` exists but is not deployed, returning 404 from Vercel.
- iOS invite-link generation in `ios/Sources/App/InviteLink.swift` produces a URL that does not match the deployed web route shape.

## Fix scope

- Identify the actual failure point (AASA vs Vercel deploy vs URL mismatch) and fix it.
- Add a CI regression test that prevents recurrence. Two pieces:
  - **Synthetic curl check** — CI job hits the live deployed invite URL with `curl -I`, asserts non-404 response.
  - **AASA validator check** — CI job hits Apple's AASA validator endpoint (or an equivalent live validation) for the deployed `apple-app-site-association` file, asserts the `/join/*` path is registered.
- These checks belong in the single `ci.yml` workflow as new jobs per [[../../../../home/node/.claude/projects/-workspace/memory/feedback_ci_lane_shape|CI lane shape convention]].

## Acceptance criteria

- [x] Tapping a freshly generated invite link on a real iOS device opens either the iOS app (via Universal Link) or the web fallback at `/join/<roomId>`, never a 404. _(Root cause was the Vercel framework-preset misconfig already fixed in 01eca6f as part of TB-16: `gettoit-web` had been created with Framework Preset "Other" rather than "Next.js", so the AASA file served from `web/public/` while every Next.js route 404'd. Live verification: `curl -I https://gettoit.app/join/<id>` now returns HTTP 200 with `x-matched-path: /join/[roomId]` and `x-powered-by: Next.js`. AASA still serves `/join/*` and is ingested by Apple's CDN at `app-site-association.cdn-apple.com/a/0.1.0/gettoit.app`.)_
- [x] CI fails when the deployed invite URL returns a 4xx/5xx (synthetic curl test). _(New `invite-link-canary` job in `.github/workflows/ci.yml`: runs after the `web` lane on every PR + push to main, curls `https://gettoit.app/join/ci-canary-bug-01-regression-test`, fails on 404 or any 5xx.)_
- [x] CI fails when AASA validation fails (path missing, JSON malformed, content-type wrong). _(New `aasa-validator` job in `.github/workflows/ci.yml`: runs after the `web` lane in parallel with the canary, asserts (a) HTTP 200, (b) `content-type: application/json`, (c) valid JSON body, (d) `/join/*` registered in `applinks.details[].components[]`. Second step asserts parity with Apple's CDN copy so an Apple-side ingestion failure is also caught.)_
- [ ] Manual smoke check: paste invite link into iMessage on real device, tap it — destination loads.
