# bug-11 — Fixture factories ship fictitious venue names in the app target

## Problem

`VerdictScreen.swift`, `CheckinScreen.swift`, and `LockedScreen.swift` define
their display data structs (`Verdict`, `Plate`) with `static func fixture()`
factories — `fixture()`, `soloFixture()`, `noSurvivorFixture()` — that build
fully-populated sample instances with hardcoded fictitious content (place names
`Pico's Taqueria`, `Ren Soba`, `Café Lou`, `Halal Cart`, sample rule text, etc.).

These factories live in `ios/Sources/App/` — the **shipped app target** — not
the test target. The strings are compiled into the production binary as dead
code.

## Surfaced by

tb-26 (remove fictitious fallback venues). tb-26 deleted `QuizDummyCandidates`,
which was *reachable from a live code path* — the Q5 candidate fetch genuinely
returned it and rendered it to real users. These `fixture()` factories are the
**non-reachable cousins**: the same "hardcoded fictitious venue names in the app
target" smell, but strictly weaker.

## Not user-facing

The factories are `static func`s called only by snapshot tests and SwiftUI
`#Preview` blocks. No production path calls them — the real `VerdictStore`
populates `Verdict` from the database. A user cannot reach them. Risk today is
near-zero; this is a hygiene / tech-debt item, not a defect with user impact.

## Suggested fix (for triage to confirm)

Move the `fixture()` family into the test target (or a preview-only,
non-shipped file) so the app binary carries zero fictitious venue strings.
Wrinkle: the `#Preview` blocks in the screen files likely depend on these
factories — relocating them needs the previews to still resolve, so it is a
small but real change, not a one-liner.

## Acceptance criteria (tentative — triage to finalize)

- [ ] No fictitious venue names remain in the `ios/Sources/App` app target.
- [ ] Snapshot tests and SwiftUI previews still build and pass.
- [ ] The `ios` CI lane is green.

## Agent Brief

**Category:** bug
**Summary:** Move the verdict/check-in/locked snapshot-test fixture factories off the shipped iOS app target so the production binary carries no fictitious venue strings.

**Current behavior:**
The display-model value types for the verdict, check-in, and locked surfaces — `Verdict`, and the `Plate` type(s) used by the check-in and locked screens, each nested inside its `*Screen` view — expose `static func` fixture factories: `fixture()`, `soloFixture()`, and `noSurvivorFixture()`. These build fully-populated sample instances from hardcoded fictitious content: place names (`Pico's Taqueria`, `Ren Soba`, `Café Lou`, `Halal Cart`), sample rule text, and sample receipts. The factories are declared inside the `GetToIt` app target, so those strings compile into the shipped production binary as dead code.

No production code path calls them — the live `VerdictStore` populates `Verdict` from the database. Every caller is in the test target, which imports the app module with `@testable import GetToIt`. There are no SwiftUI `#Preview` blocks anywhere in the app sources, so nothing in the app target depends on these factories. (The original issue body speculated that previews depend on them — that is not the case; the only dependents are test files.)

**Desired behavior:**
The fixture factories live in the test target only. The app target compiles zero fictitious venue strings. The factories stay callable from every existing test file at the same call sites (`VerdictScreen.Verdict.fixture()`, the `Plate` `.fixture()` calls, etc.), so relocating them requires no edits to the test files that consume them.

**Key interfaces:**
- The `Verdict` value type — its `fixture()`, `soloFixture()`, and `noSurvivorFixture()` factories move into a test-target `extension`.
- The `Plate` value type(s) used by the check-in and locked screens — their `fixture()` factory(ies) move into a test-target `extension`.
- `@testable import GetToIt` already grants the test target visibility into the app target's types and their memberwise initializers, so a test-target extension can reproduce each factory body unchanged.
- The factories are currently declared `public`; once test-target-only they no longer need `public` access.

**Acceptance criteria:**
- [ ] No fictitious venue names (`Pico's Taqueria`, `Ren Soba`, `Café Lou`, `Halal Cart`) or their associated sample rule/receipt strings remain anywhere in the `GetToIt` app target sources.
- [ ] `fixture()` / `soloFixture()` / `noSurvivorFixture()` resolve from every existing test file with zero changes to those test files' call sites.
- [ ] iOS snapshot/smoke tests still build and pass.
- [ ] The `ios` CI lane is green.

**Out of scope:**
- The `noSurvivorFixture()` "No spot fits" hero copy is a genuine no-survivor UI string, not fictitious venue content — it must keep working but is not in scope to scrub.
- No change to production verdict/check-in/locked rendering, to `VerdictStore`, or to how real data populates these types.
- No marketing-branding copy pass on the fixture strings (they carry `placeholder: marketing-branding pass` markers; that is a separate track).
- Fixtures that already live in the test target are not in scope.
