# GetToIt 0.1.0 â€” Product Requirements Document

## Problem Statement

A small group of friends (typically 2â€“6) decides on a trivial-to-mid going-out activity â€” most commonly "where to eat tonight." The group has rough intent ("dinner") but stalls before committing. Existing tools fail in three concrete ways:

- **iMessage polls** require someone to pre-curate options, and that curator becomes the de-facto decider.
- **Google Maps / Yelp** is one person's screen â€” no shared answer, no group fairness.
- **ChatGPT** is a solo dialogue, not a community ritual.

The result is the canonical "where do you wanna eat? / I don't care, you pick" loop, which produces three distinct failure modes:

1. The group never decides â€” the night dissolves.
2. The group nominally decides but doesn't follow through â€” somebody defects on the way to the door because the rejected option becomes salient post-decision.
3. The de-facto decider over-indexes their own preference, and the loser carries silent resentment into future decisions.

This is **decision paralysis** in its group-amplified form. The product target is the gap between *agreed verdicts* and *followed-through verdicts*.

## Solution

A native iOS app (with a web fallback for non-installers) that runs a 60-second group decision ritual:

- The initiator opens the app, picks the food vertical, and shares a Universal Link in their group chat.
- Invitees tap the link and answer a fixed 5-question quiz in parallel, anonymously, in under 60 seconds each.
- The app pulls candidate restaurants from Foursquare (with Apple MapKit as fallback), runs an Elimination-by-Aspects + Satisficing engine on the answers, and returns **one verdict** â€” a single restaurant with `where + when + who`.
- The verdict screen explains the rule that produced the pick (aggregate-rule attribution, never personal-causal), shows each member's voice was counted, and offers a pre-commit ratification button ("I'm in") before a hard-close mechanic closes the decision visibly.
- Reroll exists but costs â€” capped at 3 per session, requires a stated reason, the reason becomes a new constraint, and the reason is visible to the group.
- The next day, a push notification asks each member "Did you go?" â€” feeding the north-star metric (% of verdicts followed-through) which determines whether the thesis is real.

Three design choices are non-negotiable and structural to the product working at all:

1. **Algorithm-as-decider with named rules** â€” never "Maya said no," always "Budget cap cut Sushi Ren." Drawn from the procedural-justice synthesis ([[../50_product/verdict-screen-spec]]).
2. **NEED-then-EQUALITY distribution** â€” hard-constraint vetoes (dietary, allergy, budget) prune first; soft preferences balance across sessions invisibly; equity framing ("your turn") is actively suppressed.
3. **Hard close after ratification** â€” the verdict visibly closes. Without this mechanic, the group says "sounds good" and nobody goes.

## Implementation Decisions

### Stack and platform

- **iOS-only native client in 0.1.0**, Swift + SwiftUI, minimum deployment target iOS 17. Locked in [[../60_engineering/adr/0001-ios-tech-stack-supabase|ADR 0001]] and [[../60_engineering/adr/0008-ios-min-target-17|ADR 0008]].
- **Backend on Supabase** â€” Postgres + Realtime + Auth + Storage + Edge Functions. Single data plane. Locked in ADR 0001.
- **Places data** â€” Foursquare primary, Apple MapKit POI search as iOS-side fallback. Edge Function proxy + `places` cache table in Supabase. Locked in [[../60_engineering/adr/0002-places-data-foursquare-mapkit|ADR 0002]].
- **Telemetry** â€” Supabase tables + SQL views (`metric_follow_through_pct`, `metric_time_to_verdict_p50`, `metric_invite_acceptance`). No third-party analytics in 0.1.0. Locked in [[../60_engineering/adr/0005-telemetry-supabase-event-store|ADR 0005]].
- **Privacy posture** â€” claimed-retained, anonymous-30-day TTL, in-app delete with cascade, no third-party preference sharing, US-only beta. Locked in [[../60_engineering/adr/0006-privacy-posture-0.1.0|ADR 0006]].
- **Auth** â€” anonymous default + post-quiz Sign in with Apple upgrade chip on Waiting surface. Locked in [[../60_engineering/adr/0007-auth-anonymous-default-apple-upgrade|ADR 0007]].

### Modules

0.1.0 ships as 11 deep modules. Each encapsulates a focused responsibility behind a stable interface and is testable in isolation.

#### iOS â€” `ios/`

1. **SessionStore** â€” input: a `room_id`. Output: a reactive snapshot of room state (members, votes, verdict, status) for SwiftUI views. Internals: SwiftData mirror + Realtime Broadcast subscriber + optimistic-write reconciliation against Supabase. Conflict resolution: last-write-wins by `created_at`. Hides realtime + offline complexity from views. The deepest module in the iOS app.

2. **AuthCoordinator** â€” input: a desired auth state (`anonymous`, `linked-apple`). Output: a current `Auth` object with `user_id` and identity status. Internals: wraps `supabase-swift` `signInAnonymously()`, `signInWithIdToken()`, and `auth.link_identity()` plus the `Authentication Services` framework call for Apple. Guarantees that anonymous-to-claim merges never lose data â€” this is the failure mode most fatal to product trust.

3. **InviteLink** â€” input: a `room_id`. Output: a `https://gettoit.app/join/{roomId}?inviteToken={token}` Universal Link string + share-sheet trigger. Also parses incoming links into `(roomId, inviteToken)` pairs. Pure functions on top of `URLComponents`. Universal Links are configured via the `Associated Domains` entitlement + a server-hosted `apple-app-site-association` file.

4. **TimerCoordinator** â€” input: an initiator-set duration (5 / 10 / 15 / 30 minutes; default 10). Output: a reactive countdown + a `fire()` method that triggers the verdict computation server-side. Drives both the auto-fire path (timer hits zero) and the manual-fire path ("Decide now" button). Enforces the minimum quorum of 2 answers (initiator + 1 other) before allowing fire. Pure timer + intent; the actual verdict computation lives server-side.

5. **PushCoordinator** â€” input: a permission request trigger (post first "I'm in" ratification per session). Output: registered APNs device token written to the user's row. Internals: `UserNotifications` framework + `UIApplication.registerForRemoteNotifications`. Pairs with the Edge Function APNsSender for delivery. Owns the pre-permission copy timing on the Verdict surface.

6. **TelemetryWriter** â€” input: event records (`event_type`, optional `room_id`, optional `user_id`, `properties` payload). Output: rows in Supabase `events` table. Thin wrapper. RLS policies enforce per-user write scope. Check-in rows go to a separate `check_ins` table per the schema in ADR 0005.


#### Edge Functions â€” `supabase/functions/`

8. **PlacesProxy** â€” input: `(geo: { lat, lng }, radius_meters, filters: { dietary[], price_tier, open_at })`. Output: an array of candidate options shaped to the `options` table. Internals: queries the `places` cache in Supabase first (keyed by `geo_h3` + `query_signature`, TTL 24h hot / 7d cold), falls back to Foursquare API (proxied with the API key from env), writes new results into the cache. iOS calls the MapKit fallback directly on its own when Foursquare returns thin results or fails (per ADR 0002); the Edge Function itself never calls MapKit. Hides Foursquare quirks behind a clean filter interface.

9. **APNsSender** â€” input: `(user_ids[], notification: { title, body, payload })`. Output: APNs delivery receipts. Internals: signs a JWT with the `.p8` key from env, posts to APNs HTTP/2 endpoint. Called from two triggers: (a) a Postgres webhook on `verdicts INSERT` for the "verdict is in" notification, (b) the CheckinScheduler cron job.

10. **VerdictEngine** â€” input: a `room_id`. Output: a `verdicts` row + a set of `option_cuts` rows + a broadcast on the `room:{roomId}` channel. Internals: runs the locked engine â€” EBA pruning chain on Q1â€“Q4 vetoes, soft-pref silent relax (`cuisine veto â†’ vibe floor â†’ radius widen`) on empty-survivor, hard-need-veto preservation (dietary / allergy / budget never relax), regret-of-omission scalar summed across members as tiebreaker, random fallback on flat regret signal, terminal no-survivor state when even after relax the set is empty. The single canonical implementation lives server-side; iOS never recomputes â€” it reads the result. Server-authoritative by design ([[../60_engineering/stack-patterns|stack-patterns.md]] Â§"Deadline / quorum / verdict computation"). Implementation choice â€” pure SQL with a trigger, or TypeScript in an Edge Function â€” is left to the engineer; the interface is the contract.

11. **CheckinScheduler** â€” input: time tick (every minute via `pg_cron`). Output: enqueued APNs sends for `verdicts` rows whose `computed_at` is 12â€“24 hours ago and that have not yet been notified. Records the notification dispatch so it never fires twice per verdict. Pairs with the Check-in surface (S08) which records the response into `check_ins`.

#### Schema and data layer

Tables and approximate columns (full migrations defined separately):

- `rooms (id uuid, creator_user_id uuid, deadline_at timestamptz, status text, vertical text, radius_meters int, timer_minutes int, created_at)` â€” `status` âˆˆ {`open`, `verdict_ready`, `locked`, `expired`}.
- `members (room_id uuid, user_id uuid, role text, joined_at timestamptz)` â€” `role` âˆˆ {`owner`, `participant`}.
- `options (id uuid, room_id uuid, place_id text, payload jsonb)` â€” `payload` carries Foursquare-shaped restaurant data.
- `votes (room_id uuid, user_id uuid, q1_vetoes text[], q2_budget int, q3_walk_minutes int, q4_vibe int, q5_regret jsonb, created_at)` â€” one row per (room, user). `q5_regret` is `{option_id: regret_score}`.
- `verdicts (room_id uuid, option_id uuid, computed_at timestamptz, method text, rule_text text)` â€” `method` âˆˆ {`quorum`, `deadline`, `manual`, `no_survivor`}.
- `option_cuts (verdict_id uuid, option_id uuid, cut_reason text, cut_text text)` â€” feeds the Cuts drawer.
- `places (geo_h3 text, query_signature text, payload jsonb, cached_at timestamptz)` â€” Foursquare cache.
- `events (id uuid, room_id uuid null, user_id uuid null, event_type text, properties jsonb, created_at)` â€” telemetry.
- `check_ins (room_id uuid, user_id uuid, outcome text, reason text null, created_at)` â€” `outcome` âˆˆ {`went`, `skipped`, `snoozed`, `no_signal`}.

**Row Level Security** â€” every table has a `WHERE room_id IN (SELECT room_id FROM members WHERE user_id = auth.uid())` style policy. `places` is read-shared and writeable only by the Edge Function service role. `events` and `check_ins` are insert-only per user.

**Verdict trigger** â€” `AFTER INSERT ON votes` fires the VerdictEngine when `count(votes WHERE room_id = NEW.room_id) >= 2` AND `(rooms.status = 'open')` AND either `now() >= rooms.deadline_at` OR the initiator marked `status = 'firing'` via a manual-fire RPC. Cron job runs every minute scanning `rooms WHERE status = 'open' AND deadline_at <= now()` for the auto-fire path.

#### Realtime patterns

- **Realtime Broadcast** carries live vote events and the `verdict_ready` notification. Subscribers receive in 50â€“200 ms.
- **Presence** carries "who's in the room" on the Waiting surface.
- **Postgres Changes** is reserved for cold-start hydration only â€” never the live-fanout channel. Per stack-patterns Â§"Realtime".

### Mechanics â€” engine specifics

The engine ([[../50_product/framework-comparison|framework-comparison]] Â§"The hybrid engine") runs in this order:

1. **Q1â€“Q4 EBA pruning chain** â€” each member's answers produce per-attribute vetoes / thresholds. The survivor set is the intersection of every member's survivor set. Hard-need vetoes (Q1 dietary as menu-compliance, Q2 budget cap, Q3 walk-time threshold) are non-negotiable; Q4 vibe floor and cuisine veto are soft.
2. **Soft-pref silent relax** â€” if survivors = 0, relax in order: most-cited cuisine/vibe veto first, then radius widen to the next 0.5-mile increment up to 5 mi cap, re-run pruning. Silent â€” no UI surfacing.
3. **Hard-need terminal** â€” if survivors = 0 after exhausting soft relax, exit with `method = 'no_survivor'`. The verdict surface renders the terminal "No spot fits tonight" state.
4. **Single-survivor short-circuit** â€” if exactly one option survives, that is the verdict. Skip Q5 weighting; render with the spec'd "Only one made it tonight" eyebrow.
5. **Multi-survivor tiebreaker** â€” sum Q5 regret-of-omission scalars across all members per surviving option; pick maximum. If variance is below a flat-signal threshold, fall back to random pick within the survivor set. Locked in [[../50_product/0.1.0-design-locks|0.1.0-design-locks]] Lock 3.
6. **Rule text generation** â€” produce the rule chip string by joining the active rules in plain language. Aggregate-rule attribution: "Budget cap cut Sushi Ren. Pico's had the lowest regret-of-omission." Never name a person. For anonymized constraints, use attribute attribution: "filtered for celiac-safe kitchens" not "filtered because Maya has celiac."

### Group size, fire trigger, timer

- Group size 2â€“6 is the expected range; no fixed N is enforced. Invite link is shareable; anyone in the chat can tap.
- Verdict fires on the **initiator's manual tap** ("Decide now" on the Waiting surface) OR on the **initiator-set timer expiring** (default 10 min, presets 5 / 10 / 15 / 30).
- Minimum quorum to fire = 2 answers (initiator + at least one other). If only the initiator has answered when the timer expires, the room enters `status = 'expired'` with no verdict.
- After fire, the invite link's join action is disabled. Late tappers route to the read-only verdict surface.

### Identity edge cases

- Anonymous identity is device-bound. Reinstall = new anonymous identity. Same human on a second iOS device = separate anonymous identity. Cross-device requires Apple link.
- Web-fallback voters are anonymous indefinitely; they cannot link Apple from the browser. A web user who later installs the app lands as a fresh anonymous identity â€” no historical merge. Flagged as a v2 feature in ADR 0007.

## Testing Decisions

### What makes a good test

- **Test external behavior, not implementation details.** A test should fail when the user-visible contract breaks, never when an internal refactor changes how the contract is fulfilled.
- **Fixtures over mocks where possible.** For the VerdictEngine especially, a fixture is a `(room state, votes[]) â†’ expected verdict` tuple. Mocks invite implementation coupling.
- **Integration tests hit a real database.** A throwaway Supabase project (CI-spawned) is the test target for SessionStore, AuthCoordinator, and the schema. Mocked-DB tests in this domain catch the wrong bugs.
- **One assertion per scenario when reasonable.** A test should name the behavior, not enumerate properties.
- **No test couples to a private function.** If you can't drive a code path through the public interface, the interface is missing.

### TDD discipline

0.1.0 is built TDD. For every module below, the loop is: write a failing test against the module's public interface â†’ implement the smallest code that passes â†’ refactor with the tests as the safety net. No code lands without a test that would have failed against the previous main.
