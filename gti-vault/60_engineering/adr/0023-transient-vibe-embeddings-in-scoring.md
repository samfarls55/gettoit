# 0023 -- Vibe embeddings run transiently in recommendation scoring

## Status

Accepted -- 2026-06-11. Decision from the embeddings scoring grill.

## Context

ADR 0022 moves candidate-pool data to Google Places API and draws a conservative retention boundary: Google place display content, atmosphere fields, summaries, raw payloads, provider facts, and provider-fact component scores are transient. Vibe is still load-bearing for Q5 and verdict scoring, and Google `reviewSummary`, `generativeSummary`, and atmosphere fields can help place candidates on the app's 1-5 Vibe band.

The obvious implementation trap is to treat embeddings as part of the Google provider layer or as a durable derived place profile. That would blur provider fetching with GetToIt deciding, and it would make summary-derived vibe positions look app-owned enough to store.

## Decision

Vibe embeddings run in the recommendation / scoring layer, not as a durable Google provider cache or as client logic.

The Google provider layer fetches current Places data under server-owned field masks, budgets, attribution, and redaction. The recommendation / scoring layer receives only the transient candidate fields it needs, assembles transient **Vibe evidence**, compares it to versioned app-owned **Vibe anchors**, produces a transient **Vibe fit signal**, and discards the Google-derived text, embeddings, and `vibe_position` after Q5 card generation or verdict scoring.

Q5 probe fetches and final verdict fetches may request Google `reviewSummary` / `generativeSummary` only through internal scoring field-mask versions when Vibe fit is enabled and needed for that path. Summary fields remain server-only, are discarded before any response or storage, and are omitted from display-only / refetch masks or paths where embeddings are disabled, budget-skipped, or otherwise unnecessary where provider field-mask control allows.

The Vibe fit module receives a narrowed internal `VibeFitCandidate`-style DTO, not raw Google Places responses. That DTO may carry the in-flow candidate identifier, Google Place ID for joining under ADR 0022 boundaries, Plan meal-time context, allowed weak structured hints, server-only summary text when enabled, and minimal provider / source metadata needed for controlled receipts. It must not expose unrelated provider facts such as cuisine / type, price, rating, service format, full address, display name, reviews, or raw payload fields to vibe extraction / projection.

Vibe embeddings run only after hard eligibility cuts. The backend must not spend embedding calls on candidates already disqualified by search area, price cap, meal-time eligibility, service mode, hard dietary safety, Cuisine NEVERs, missing required metadata, or the crowd approval floor, and those hard cuts must not depend on embeddings.

One versioned Vibe fit signal feeds both Q5 card generation and final verdict scoring. Q5 and verdict may consume it differently, but they share the same app-owned vibe ruler so the member is not probed under one meaning of vibe and scored under another.

Embeddings work assumes the ADR 0022 Q5 axis language: `cuisine`, `crowd_approval`, and `vibe`. Implementation must verify that the active Google scoring seam uses those axes before wiring new Vibe fit logic; if any active seam still uses the legacy `reputation` name, embeddings work must align it rather than extending new code on the old name.

FSQ-era venue-classifier behavior is not a compatibility target for the Google embeddings path. Tests that bless Foursquare `categories`, `tastes`, price-tier vibe fallback, or pool-relative `reputation` semantics should be replaced or marked legacy / superseded when this seam is implemented.

The PRD / implementation must begin with a backend seam audit that identifies the active modules for Google candidate payload shaping, Q5 card generation, candidate profiling, member preference scoring, summary fetch/discard behavior, and any remaining legacy FSQ / `reputation` paths. Embeddings should be wired only into the active Google scoring seam.

The canonical domain output is a continuous `vibe_position` on the 1.0-5.0 Vibe band plus confidence. Any 0-4 representation is a legacy implementation index and should be adapted at module boundaries.

For v0.1.0, Vibe anchors are hardcoded as versioned backend configuration, not stored in the database for runtime tuning. Anchor changes require code review and a scoring / anchor version bump so Q5 generation and verdict scoring stay repeatable.

Anchor phrase edits are scoring changes even when implementation code is unchanged. Any anchor edit must bump the Vibe anchor / scoring version and be covered by fixture expectations so old and new Q5 / verdict behavior remain explainable.

The client sends stable Vibe band identity, not scoring text. Backend scoring owns the mapping from the five ordered UI labels to 1-5 band positions and app-owned anchor phrase sets, so UI copy changes do not silently change recommendation behavior.

The PRD should inventory the current five Vibe UI labels, define their stable backend IDs / 1-5 positions, define a v1 starter anchor set, and cross-check the anchors against those UI labels. Anchor phrase sets may be richer than the UI labels; for example, a high-end Rowdy band may include "energetic", "loud", "packed", "high-energy", "party-like", and "scene-y" so scoring does not depend on the single word "Rowdy".

Anchor projection uses a weighted centroid across the five Vibe bands, not nearest-anchor-only classification. Clear top-anchor wins still land near that band, adjacent ambiguity may land between bands, and non-adjacent conflict lowers confidence rather than pretending high precision.

Vibe confidence combines evidence amount, anchor clarity, and conflict penalty. More valid atmosphere spans raise confidence up to a cap; clear anchor separation raises confidence; non-adjacent span conflict lowers confidence unless explicit meal-time evidence resolves it. Venue type/category, price, rating, and crowd approval evidence must not boost vibe confidence.

The PRD should define the v1 formula contract rather than leave projection / confidence as opaque implementation detail. The shape is cosine similarity from selected evidence spans to anchor phrases, aggregation of anchor scores by Vibe band, weighted-centroid projection to `vibe_position`, confidence from evidence amount plus anchor separation minus conflict, and distance-based member vibe scoring blended / capped by confidence. Exact thresholds and constants may be v0.1.0 defaults tuned during implementation with fixture evidence before merge.

Formula constants should live in a versioned backend config object, such as `VIBE_FIT_CONFIG_V0_1_0`, rather than as scattered literals. The config should include anchor version, max spans, confidence caps, weak-hint weights, timeout / budget names, and formula constants so durable scoring-version receipts can identify the exact app-owned ruler used.

For v0.1.0, the embedding model is Voyage `voyage-4-lite` with its default 1024-dimensional output, called from the backend in batches where practical. This was chosen as the best value fit for short atmosphere-span matching against app-owned Vibe anchors: current provider docs list it at the same $0.02 / 1M-token price point as OpenAI `text-embedding-3-small`, with a newer embedding-family profile, smaller default vectors, and a free-token tier. The model choice is locked for the first implementation so fixtures, scoring versions, and cost guardrails have one target.

Supabase Edge Functions call Voyage through a thin backend wrapper module that owns API URL, auth header handling, request / response parsing, batching, timeout, retry, and error mapping. Scoring code calls the wrapper's vibe-text embedding function rather than scattering provider calls. v0.1.0 does not introduce a generic multi-provider embedding framework.

Most Vibe fit behavior should live in a separate pure backend module: span assembly, anchor config, projection, confidence, and conversion to member-facing vibe axis score. Q5 and verdict handlers orchestrate calls into that module; they should not own embedding provider details or duplicate vibe math. Implementation should expose one shared core `computeVibeFit`-style scorer and thin Q5 / verdict adapters, so both use the same vibe ruler while preserving Q5-specific confidence and contrast rules.

Because Vibe anchors are app-owned text, their embeddings may be precomputed or cached as app-owned configuration if that becomes useful. The v0.1.0 default is simpler: batch the app-owned anchor phrases together with transient candidate spans during the active scoring flow. This allowance does not extend to Google-derived candidate spans or their embeddings.

Within one active scoring flow, the backend may reuse a transient in-memory `place_id -> Vibe fit signal` result so Q5 card selection and verdict scoring do not embed the same candidate repeatedly. That reuse is not a cache: it must not survive the request / deterministic scoring cycle, cross Rooms, or be written to durable storage.

## Consequences

- The embedding path must not persist Google-derived source spans, summary text, atmosphere text, embeddings, `vibe_position`, numeric vibe confidence, or provider-fact component scores.
- Durable records may keep only final aggregate Verdict fit score, version IDs, Google Place IDs where ADR 0022 allows them, Q5 outcome labels, axis roles, and app-owned controlled receipt / reason codes. The durable scoring version should identify the Google scoring field-mask versions, Vibe anchor version, span assembler version, embedding model/provider, projection / confidence formula version, Q5 generation rules version, and verdict scoring formula version.
- The PRD / implementation should seed controlled vibe receipt codes such as `vibe_no_evidence`, `vibe_low_confidence`, `vibe_conflicting_evidence`, `vibe_mealtime_weighted`, `vibe_embedding_unavailable`, `vibe_embedding_budget_exhausted`, `vibe_embeddings_disabled`, `selected_vibe_low_confidence_relaxed`, `selected_vibe_contrast_relaxed`, `selected_vibe_high_confidence_keep`, and `selected_vibe_high_confidence_drop`. Released durable code meanings remain append-only and versioned under the existing **Reason codes** rules.
- Schema changes are allowed only where the current durable receipts / scoring-version structures cannot cleanly represent those app-owned controlled codes and version components. v0.1.0 should not add vector tables, venue vibe profile tables, stored confidence / position columns, stored extracted phrase tables, or training-data capture tables for Google-derived text.
- Any move away from Voyage `voyage-4-lite`, including to Voyage `voyage-4`, OpenAI `text-embedding-3-small`, or another provider, requires fixture / eval evidence that the locked model fails the vibe/type boundary or anchor projection quality bar.
- Durable embedding learning is deferred. v0.1.0 does not create durable venue vibe profiles, stored summary-derived embeddings, stored extracted vibe phrases, cross-room Place ID to vibe-position caches, pgvector / Supabase Vector tables, or training / eval datasets containing Google-derived spans.
- In-memory dedupe of Vibe fit signals is allowed only inside one active Q5 generation or verdict scoring flow.
- Missing or low-confidence vibe evidence weakens the vibe component toward neutral; it does not disqualify a candidate. Candidates with no vibe evidence remain eligible to win the final verdict if they score strongly enough on other axes, but they do not receive a strong positive vibe match from missing evidence. The final vibe-axis formula should blend toward neutral and cap the maximum positive contribution of low-confidence vibe fits so sparse evidence cannot accidentally produce a perfect vibe match.
- Q5 card generation should prefer high-confidence Vibe fit signals for the vibe-axis card, with a stronger confidence bar than final verdict scoring. Vibe-keep candidates should sit near the member's target Vibe band; vibe-drop candidates should be meaningfully outside the target band while still matching the other axes, not forced to cartoonish extremes unless the pool requires it. If the pool is thin, Q5 may relax vibe confidence or contrast distance before failing, using app-owned controlled reason codes rather than storing provider-derived evidence. Final verdict scoring can tolerate neutral / low-confidence vibe candidates because it aggregates many axes and members.
- Vibe evidence is embedded at the extracted-span level and then combined into one transient Vibe fit signal. Strongly conflicting span results lower confidence; explicit daypart / meal-time clues may weight spans toward the Plan's meal time.
- Quality / value adjectives such as "excellent", "best", "favorite", "popular", and "highly rated" are excluded from vibe evidence unless the phrase also contains actual atmosphere texture. "Popular" is crowd approval; "packed", "crowded", "busy", "buzzy", or "hard to hear" may be vibe evidence when they describe experienced atmosphere. "Local" / "neighborhood" is weak vibe context only when paired with stronger atmosphere wording or "feel" / "spot" framing.
- Price / service facts such as "expensive", "cheap", "budget-friendly", "counter-service", "takeout", and service-format uses of "fast-casual" are excluded from vibe evidence. Atmosphere uses of "upscale", "elegant", "romantic", "date-night", "white-tablecloth", and mood-facing "casual" may be vibe evidence.
- The local span assembler should handle simple negation such as "not loud", "not too crowded", and "not quiet" so the signal is not inverted. Complex negation, sarcasm, or ambiguous phrasing should lower confidence or be excluded rather than handed to scoring as certain evidence.
- Google "Atmosphere" is a provider SKU tier, not GetToIt's vibe ontology. Structured Google fields may contribute only through a curated weak-hint allowlist, such as `liveMusic`, `goodForGroups`, `goodForWatchingSports`, and possibly weak-context `outdoorSeating`; service, meal, price, rating, menu, and dietary fields remain outside vibe evidence. Curated structured hints are not embedded as candidate text. They may apply small, confidence-capped post-embedding nudges toward compatible Vibe bands when summary-derived evidence is absent, weak, or consistent, but they must not overpower summary-derived atmosphere spans or produce high confidence by themselves.
- Google display name, address, and Maps URI are not vibe evidence and must not be embedded or used for span extraction. They remain current display content only where ADR 0022 allows them.
- Venue type facts, cuisine/menu facts, service-mode facts, dietary safety, and Cuisine NEVERs stay outside vibe embeddings so Q5's cuisine, crowd approval, and vibe axes remain separable.
- The embedding path must not infer or modify cuisine, venue type, service mode, dietary eligibility, dietary safety, Cuisine NEVER handling, price, or crowd approval later in the same flow. If extracted text contains terms such as "vegan", "steakhouse", or "seafood tower", those terms are excluded from Vibe evidence and cannot be reused as shortcuts for non-vibe axes.
- The backend owns per-Room / per-scoring-flow embedding budgets in addition to global provider guardrails. If span count, batch size, call count, timeout, quota, or cost limits are reached, remaining candidates degrade to neutral / low-confidence vibe fit with controlled receipt codes such as `vibe_embedding_budget_exhausted`.
- The PRD must specify the embedding timeout / retry policy. v0.1.0 should use one short per-flow embedding timeout, batch requests where practical, and allow at most one bounded retry only for clearly transient `429`, `5xx`, or network-timeout failures when the retry still fits the Room embedding budget. The implementation must not run per-candidate retry loops.
- Embedding inputs are batched per active scoring flow, not one candidate at a time. Q5 generation and verdict scoring should select spans deterministically, add app-owned Vibe anchor phrases, dedupe identical text within that flow, call Voyage in as few batches as the budget allows, then project candidates. Partial provider failures degrade affected candidates to neutral / low-confidence instead of starting candidate-level retry loops.
- v0.1.0 embeds at most five Vibe evidence spans per candidate. Span selection must be deterministic by source priority, atmosphere signal strength, and meal-time relevance; extra valid spans may affect only aggregate provider-content-free metrics or controlled receipt buckets, not durable text or vectors.
- Embedding provider failure, timeout, or budget exhaustion degrades the Vibe fit signal toward neutral / low confidence rather than failing the Room. Durable records may keep only controlled operational receipt codes such as `vibe_embedding_unavailable`, not provider text or embeddings.
- Production enablement requires verifying Voyage's current API data-use / retention terms for Google-derived transient text. If Voyage's terms are unacceptable for this use, production embeddings must remain disabled or the provider decision must be revisited.
- Embeddings are controlled by a server-side feature flag / kill switch. v0.1.0 has no separate user-facing environments, so rollout is a production-only checklist gate rather than a staged environment rollout: keep embeddings disabled until `VOYAGE_API_KEY`, Voyage retention review, fixtures, privacy tests, and a smoke test are complete, then enable with the kill switch still available. Clients do not control this flag. Disabled embeddings skip Voyage calls but must not fall back to legacy category/type vibe heuristics. The Vibe fit module may still use curated weak structured hints, otherwise candidates degrade to neutral / low-confidence vibe fit. Q5 should still attempt to preserve the vibe-axis card shape when embeddings are disabled, but any weak-hint or neutral contrast must be treated as relaxed / low-confidence through controlled reason codes rather than as a high-quality vibe probe.
- Observability may record aggregate, provider-content-free metrics such as candidate counts with no vibe evidence, selected / embedded span counts, embedding calls per flow, Voyage latency, timeout / error / budget counts, aggregate confidence buckets, aggregate anchor / position buckets, Q5 vibe relaxation counts, and no-results counts due to vibe confidence or contrast shortage. Logs and analytics must not carry source spans, summaries, vectors, per-place `vibe_position`, place names, provider facts, or broad-stream Google Place IDs.
- Implementation is not complete without golden fixtures covering vibe/type separation, quality/crowd/service leakage, meal-time conflicts, synonym coverage across all five Vibe anchors, low/no evidence behavior, embedding unavailable / budget-exhausted fallback, and no durable leakage of spans, vectors, or `vibe_position`. Fixtures must test both span extraction and final Vibe fit behavior so venue type, crowd approval, price, and service facts cannot leak back into `vibe_position` after extraction. The PRD should also define lightweight mapping-quality acceptance criteria: synonym-heavy fixtures for each band project within expected ranges, type / food / service-only summaries land neutral / low-confidence, mixed quiet / lively evidence lowers confidence, and Q5 vibe keep / drop fixtures choose the intended contrast.
- The quality gate may block merge if the fixture suite passes but a narrow manual calibration pass shows common vibe language clearly landing on the wrong part of the Quiet-to-Rowdy band. Manual calibration may use app-authored examples and transient real Google observations, but not stored Google-derived summaries or spans. v0.1.0 should tune anchors / formula against this gate without turning the work into open-ended model perfection.
- Security / privacy tests must cover the Voyage path: request text and response embeddings are not logged, provider errors are redacted, `VOYAGE_API_KEY` never appears in logs or thrown errors, and timeout / error receipts carry only controlled codes, status class, step, and latency.
- Before enabling embeddings in production-only v0.1.0, the PRD should require a short manual privacy / logging audit in addition to automated tests. Inspect logs for one successful embedding run, one provider failure, one budget exhaustion, and one disabled-mode run, confirming no summaries, extracted spans, vectors, place names, numeric `vibe_position`, numeric confidence, or other provider-derived internals leak into logs, analytics, receipts, or client responses.
- The PRD must require a small synthetic, app-authored eval / fixture suite with realistic restaurant-summary shapes that cover extraction and projection behavior without copying provider text. Normal CI uses deterministic fake embeddings and synthetic, app-authored summaries. The fake embedder should use fixed vectors by fixture label / deterministic embedding map rather than a local mini semantic model, so tests exercise extraction, projection math, and integration behavior with explainable failures. Voyage integration tests are mocked by default; any live Voyage smoke test must be secret-gated and explicitly enabled, for example with `VOYAGE_API_KEY` plus `RUN_LIVE_EMBEDDING_SMOKE=1`, so CI does not depend on provider drift, quota, or cost. Real Google summaries may inform transient manual analysis, but copied Google summary text or extracted Google spans must not be committed into repo fixtures, vault notes, or durable eval datasets.
- Implementation may include an opt-in local calibration script that sends only app-authored synthetic summaries and Vibe anchors to Voyage, then prints projected positions / confidence for developer review. It must require `VOYAGE_API_KEY`, stay outside normal CI, and write no persistent outputs unless a developer explicitly redirects stdout.
- Local development may use a deterministic fake embedder behind an explicit dev/test flag or degrade to neutral when `VOYAGE_API_KEY` is absent. Production must not silently use fake embeddings.
- v0.1.0 skips POS-tagging dependencies and LLM span classification unless fixture tests prove local span assembly cannot protect the vibe/type boundary.
