# 0022 -- Places data: Google Places API primary

GetToIt will switch the candidate-pool provider entirely from Foursquare / MapKit to the Google Places API. The old cost-floor decision in ADR 0002 no longer fits the product: vibe is load-bearing for Q5 and the verdict experience, and Google Places Enterprise + Atmosphere gives the structured atmosphere fields plus AI-powered summaries the product needs.

Durable app records store only app-owned state and provider identity: `place_provider = "google"`, Google Place ID, room / Plan / verdict IDs, votes, outcome labels, winner / reroll metadata, the ranked verdict slate of up to four Google Place IDs, final aggregate fit scores tied to the Room / verdict and scoring version, decided timestamp, app-owned receipts / cut reasons, and fully app-authored verdict metadata. Persisted rule text must be place-name-free unless it is fully app-authored without Google display content; rule copy that includes the place name is composed at render time after a current Place ID refetch. Active Room storage follows the same boundary: it may keep Google Place ID, but not full Google display payloads or provider-derived candidate facts such as place name, price level, rating, types, hours, atmosphere fields, summaries, or distance. Q1-Q4 should be shaped as provider-aligned request constraints used to fetch Google places, so those facts remain transient API inputs rather than durable engine facts. Google place display content is transient: Q5, live verdict, and read-only verdict surfaces refetch current Google display data by Place ID instead of replaying cached Google display content.

For future retention choices, GetToIt uses a learning-first policy inside the Google boundary: retain app-owned learning data when it is useful for later scoring, analytics, and model improvement, and choose the structure that best fits the use case. Valid forms include structured outcome labels, reason codes, aggregate telemetry, versioned receipts, final aggregate fit scores, and Google Place ID references where the interaction specifically needs the Place ID. This principle never permits durable storage of Google place display content, raw provider payloads, provider facts, or provider-fact component scores.

The degraded historical verdict record is app-owned only: Plan name, decided date/time, "Place unavailable," and a short message that the verdict's place details are no longer available. It does not show a stale place name.

The provider migration intentionally reopens older quiz/fetch separation decisions. Q1-Q4 may move toward Google fetch-time constraints where the member is expressing a true requirement: Search area, meal/open timing, price cap, dine-in/takeout fit, dietary hard needs, and Cuisine NEVERs. Positive cuisine craving and vibe preference stay softer scoring / probe signals. For group verdicts, vibe targets are not intersected or enforced as hard provider filters; if the provider request needs a vibe-related shape, it should use a broad room-derived band inside the server-owned call budget.

Q5 keeps the old three-card strict-factorial interaction shape, but its axes are remapped to Google-native value: cuisine, crowd approval, and vibe. Vibe is the most expressive axis; crowd approval replaces the older Foursquare-era reputation axis. In v0.1.0 UI, Q5 cards show place names only and do not display Google summary text. Q5 uses per-member probe fetches to learn preference weights; it is not required to be the same candidate pool that the final verdict ranks. When a member selects exactly one Q1 craving chip, Q5 should show two cards from that selected cuisine mapping and one contrast card outside it: two cards keep the cuisine axis, and the cuisine-drop card tests whether the stated cuisine actually matters. When a member selects two or three Q1 chips, Q5 still shows two cuisine-axis keep cards and one cuisine-drop / contrast card. The two keep cards use up to two selected chips chosen by member-local card-generation feasibility; any third selected chip still contributes to scoring / verdict, but Q5 does not need to display every selected cuisine because it measures axis weight rather than validating each cuisine individually. In group scoring, member count is the base cuisine-support signal, and Q5 axis-weight evidence adjusts how strongly each member's Q1 pick should matter.

The verdict compiles the group's constraints and Q5 signals into a separate server-owned verdict fetch plan. The product target for v0.1.0 is one Nearby Search request for the final verdict pool where possible, followed by GetToIt's scoring function. This ADR locks scoring inputs and invariants, not the exact scoring formula; exact weights, normalization, embeddings behavior, and formula tests belong to the future scoring-engine / embeddings design work. Because Google request limits can force implementation details, "one fetch" means one deterministic verdict fetch cycle rather than an eternal guarantee of one HTTP request; deterministic chunking is allowed only inside the server-owned call budget. If chunking is needed, all chunk results merge into one transient candidate set by Google Place ID before eligibility cuts and scoring. A place that appears in multiple chunks is still one candidate with one score; duplicate appearance is a provider artifact and does not create a boost. Chunk order, Google per-request ordering, provider ranking, and distance inside the committed Search area must not influence ranking or tie-breaking. Every candidate that survives final eligibility is scored in v0.1.0; the fetch / chunk budget should keep the merged pool bounded. If future scale requires pre-score downselection, it needs a separate explicit design, because provider order, distance inside the Search area, and duplicate chunk appearance are already rejected as ranking shortcuts. The server may transiently dedupe the final verdict fetch against Q5 probe candidates by Google Place ID during the same decision, but Q5 probe candidates are not a durable cache or guaranteed finalist list. Vibe is scored through Q5-inferred member weights against the final candidate pool rather than hard-combined at fetch time. The full scored candidate pool is transient and is not persisted as a historical archive of every scored Place ID, rank, and score. The verdict stores only a ranked slate of the top four recommendations as Google Place IDs with rank, final aggregate fit score, room / verdict IDs, winner / reroll metadata, decided timestamp, and app-owned reason / receipt metadata. Ranking sorts by final aggregate score first; in v0.1.0 only exact normalized score ties use a deterministic app-owned tie-break seed derived from the Room / verdict ID and Google Place ID, unless scoring-engine tests prove a tiny versioned epsilon is needed to prevent floating-point instability. The tie-break seed is fixed for one verdict generation: rerolls advance through the existing slate order and do not reshuffle tied entries, while a new Room / new decision gets a new seed. It does not store place names, addresses, ratings, hours, summaries, provider-fact component scores, or raw candidate payloads. Rerolls advance through the slate without a new fetch-and-score cycle, but the selected Place ID is still refetched for current display and basic availability before it is shown. If a slate entry fails current refetch or availability, GetToIt skips to the next viable slate entry without consuming an additional reroll burn; one burn is consumed only when a viable replacement recommendation is presented. If the slate is exhausted before the Plan uses all reroll burns, v0.1.0 does not mutate Room parameters or automatically start a new Google fetch-and-score cycle. The current Room has no more recommendations to offer; the user must start a new decision / Room to use different parameters. In v0.1.0 reroll reasons are app-owned feedback / receipt metadata, not new fetch constraints. If a future version wants reroll reasons to become new hard constraints, that should trigger a new decision / Room or another explicitly designed flow, because active Room parameters are locked after creation.

If a member exits before verdict, the final verdict decision set excludes them. Their Q1-Q5 answers, Q5 ratings, price cap, Cuisine NEVERs, hard dietary safety constraints, and vibe signals do not contribute to the final candidate pool or score after exit. GetToIt may keep an app-owned exit receipt, but exited members do not continue to impose hard constraints because they are no longer participating in the social plan.

If the initiator manually closes voting before every non-exited member has submitted Q5, the final verdict decision set includes only submitted, non-exited members. Not-yet-submitted members do not impose profile constraints, hard dietary safety constraints, Cuisine NEVERs, price caps, cuisine choices, or vibe signals on the final candidate pool. The Waiting surface should make clear that closing now decides for the submitted group only.

If a Q5 card's current Google display refetch fails, the candidate is dropped and the card is refilled from the current Q5 probe pool. If the system cannot form three valid Q5 cards, it first relaxes qualitative constraints, starting with vibe. Vibe is a member-selected 1-5 band, not a hard category: a selected 4 can expand to 3-5, with candidates closest to the selected value preferred. Future embeddings may make this relaxation more precise, but v0.1.0 treats it as controlled band widening. If vibe widening is still too thin, crowd approval may relax only within the server-owned quality floor: `rating >= 3.7` and `userRatingCount >= 15`. Price, meal-time eligibility, required provider metadata, and the selected Q1 cuisine mappings are not silently broadened beyond the existing keep / contrast design; if qualitative relaxation still cannot form three cards, the Q5 no-results path ends the attempt and asks the member to start a new decision rather than widening or changing the active Room's parameters.

The first Google-provider design requests Enterprise + Atmosphere for vibe, including structured atmosphere attributes plus `reviewSummary` and `generativeSummary` as transient inputs. Raw reviews and `editorialSummary` are excluded from v1. For v0.1.0 UI, summaries are not shown at all; Q5 focuses on returning the place name only. In v0.1.0 summaries are fetched to validate availability and shape, but they do not influence ranking. Missing atmosphere fields or summaries hurt vibe scoring confidence but do not disqualify a place. Missing crowd approval evidence, currently `rating` or `userRatingCount`, disqualifies a place in v0.1.0 because the app needs enough public-trust signal to recommend it. The crowd approval floor is only a minimum, not a target: above the floor, stronger combined rating / count evidence remains preferred, so a 4.6 venue with 1000 ratings can outrank a 4.8 venue with 16 ratings on crowd approval. A later engine redesign may use embeddings to match these summaries to vibe preferences, but that design must be grilled separately before any summary-derived embedding is stored or treated as app-owned data.

Verdict UI in v0.1.0 is action-minimal: show place name plus the Google Maps link and optionally formatted address. The Google Maps link is refetched or derived at render time from the current Google response by Place ID, not persisted in verdict / history. Ratings, hours, photos, summaries, atmosphere fields, and final aggregate fit score are not shown on the verdict surface in v0.1.0. The fit score is stored for audit / learning, not surfaced as a user-facing number.

Any Edge Function response containing Google display content must include an attribution payload / render instruction sufficient for the client surface. UI owns placement and styling, but cannot decide attribution is unnecessary. Degraded app-owned history that displays no Google content does not show Google attribution.

Server-side refetch may use bounded retries before taking the final app action: one immediate retry plus one short backoff for transient 429 / 5xx / network-timeout failures. Do not retry hard failures such as `NOT_FOUND`, permission or key errors, invalid field masks, or policy / config errors. Clients do not run Google retry loops; they receive the final app decision. Q5 refetch failures preserve card-order and axis-role integrity: before display, GetToIt refills same-axis cards and shuffles the final three cards once; if same-axis refill cannot be found, it regenerates the Q5 set or takes the Q5 no-results path. After any card is visible, GetToIt replaces only the failed slot with a same-axis card and preserves already-seen positions; if no same-axis replacement is possible, that member's Q5 attempt fails rather than silently changing inference shape, and the member may retry Q5 once with a fresh card set under the same locked Room parameters. Retry preserves the same three axis roles and member constraints, but creates a fresh `q5_card_set_id`, fresh deterministic shuffle seed, and fresh candidate Google Place IDs. If the retry also fails, that member's Q5 remains failed and the app stops retrying; if they exit or the initiator manually closes before retry completion, existing non-exited / submitted-member verdict rules apply. During a `decided-active` Plan, a failed current refetch for the winning Google Place ID means the winner is unavailable for action; GetToIt forces a reroll / recompute instead of sending the group to a dead place. Degraded app-owned verdict display is reserved for history / read-only contexts. Refetch failures may be durably recorded only as app-owned operational failure events: Google Place ID, room / Plan / verdict ID, failure type, timestamp, and action taken, such as `drop_card`, `replace_card_slot`, `q5_attempt_failed`, `q5_retry_started`, `q5_retry_exhausted`, `reroll_required`, `slate_entry_unavailable`, or `history_degraded`. They are not outcome labels because they describe technical availability rather than user / group reaction. They must not store failed response bodies, stale display content, or last-known place names, and they must not become a durable negative cache.

Any surface that displays Google place content without a Google Map must display Google attribution in the same visual context as that content. In v0.1.0 this applies even to name-only Q5 cards and action-minimal verdict surfaces. The Edge Function returns the attribution payload required for the specific current display response. Degraded history records that show only app-owned fallback content do not show Google attribution because they are not displaying current Google content.

**Consequences**

- ADR 0002's Foursquare primary / MapKit fallback decision is superseded. Because v0.1.0 is pre-release with no users and all tables were cleared before the provider switch, the migration should remove all FSQ / MapKit provider artifacts rather than keep dormant compatibility paths. Google Places is the only candidate-pool provider; no active schema, code path, Edge Function, config, environment variable, cache-table dependency, migration path, telemetry stream, or production fallback should depend on `fsq_*`, `fsq_place_id`, Foursquare categories, or MapKit POI fallback behavior.
- Supabase migration history for the active v0.1.0 baseline should be rewritten / squashed so a fresh DB reset does not recreate FSQ / MapKit tables, columns, functions, or provider config. Historical rationale remains in ADRs, not executable migrations. If an already-applied remote migration must remain for operational reasons, a follow-up migration must remove FSQ / MapKit artifacts before the active schema is considered valid.
- The provider migration is done only when no active FSQ / MapKit references remain in code, config, environment variables, schema, migrations, Edge Functions, tests, or telemetry; the Google Edge Function owns provider calls, field masks, attribution payloads, redaction, budgets, and guardrails; Q5 and verdict storage keep only Google Place IDs plus app-owned records; v0.1.0 UI shows only approved display fields with required attribution; history refetch and degraded states work; and tests cover no-results, refetch failure, slate / reroll behavior, guardrail failure, and storage-policy boundaries.
- ADR 0022 and `CONTEXT.md` are the authoritative provider-migration sources. Older PRDs, surface docs, and design notes may still mention FSQ / MapKit until implementation work touches them; when touched, mark those references as superseded by ADR 0022 rather than mass-editing historical docs during the grill.
- ADR 0012's meal-venue eligibility rule survives as a domain concept, but its Foursquare category-id mechanism is replaced by the Google primary-type mappings behind Q1's selectable cuisine / craving options. Q1 labels are short product-facing craving words; they do not need to match Google type names one-to-one. A member's explicit Q1 picks compile to Google `includedPrimaryTypes`; one Q1 chip may map to several Google primary types. The Q1 cap of three keeps explicit include requests under Google's type-list limit. "No preference" compiles to the union of all Q1-selectable category mappings, not every Google Food and Drink type. The floor should not use broad `includedTypes` such as `food`.
- Explicit Q1 picks do not fully trap a member's candidate pool inside those cuisines. The Google fetch planner keeps a contrast pool: selected Q1 chip mappings plus one server-owned no-preference / contrast fetch drawn from the union of Q1 mappings outside the selected chips. This preserves the Q5 cuisine-drop card and the old "stated cuisine is a strong signal, not a sealed box" behavior. A member who chooses "No preference" gets only the no-preference union pool.
- For the final group-level verdict fetch, selected Q1 chips combine by union rather than intersection. Cuisine craving remains a positive scoring signal, not a hard requirement that every recommendation match every member's selected cuisine. A No Preference member contributes no positive cuisine signal when other submitted, non-exited members selected cuisines; they are represented by the absence of cuisine pressure rather than expanding the request to every Q1 type. If every submitted, non-exited member is No Preference, the final verdict fetch uses the shared no-preference union pool. The scoring function should prefer cuisines with stronger member-count support, then adjust each member's cuisine support by Q5-inferred cuisine importance. The provider layer may include contrast / no-preference coverage only where it stays inside the server-owned call budget and provider request limits. The exact scoring formula remains deferred to the scoring-engine / embeddings design work.
- Cuisine NEVERs combine by union for group verdicts and remain hard vetoes. If any member has a cuisine NEVER, candidates matching that cuisine are vetoed for the whole group. This is intentionally the opposite of Q1 craving chips: positive cuisine cravings are soft / union / scored, while negative Cuisine NEVERs are hard / union / vetoed.
- Q2 price is a cap, not an exclusive category. A member's selected spend tier compiles to price eligibility up to that cap rather than exactly one price level. For group verdicts, the effective price cap is the strictest member cap because exceeding any member's cap breaks the hard affordability constraint. If a Google endpoint supports request-time `priceLevels`, the server sends every allowed level up to the effective cap; for Nearby Search (New), the server evaluates the cap from returned price metadata instead of treating price as a native Nearby filter. Places with missing provider price metadata are disqualified in v0.1.0.
- Meal timing is a single shared Plan / Room parameter, not a per-member quiz answer. It is hard verdict eligibility, but not necessarily a native Nearby Search filter. Nearby Search can return current / regular opening-hours fields, so the Edge Function evaluates whether a candidate appears open at the intended Plan meal time from current Google hours data. If an endpoint supports `openNow`, use it only for true immediate/current meal flows; do not pretend it answers future dinner/lunch timing. Places with missing hours are disqualified in v0.1.0, and hours are not shown in v0.1.0 UI. Changing meal time requires starting a new decision / Room rather than modifying the active Room.
- Dine-in / takeout fit is asymmetric in v0.1.0 because the downside differs by social context. For a dine-in Plan, Google `dineIn` must be explicitly true; missing or false `dineIn` disqualifies because sending a social group to a place without dine-in is a high-cost failure. For a takeout Plan, explicit `takeout: false` disqualifies, but missing `takeout` does not because most meal venues can support some form of takeout and uncertainty is lower-risk.
- The v0.1.0 Q1 craving chips are: American, Mexican, Italian, Japanese, Chinese, Thai, Indian, Mediterranean, Middle Eastern, Korean, Vietnamese, Seafood, and Comfort Food. Vegan is excluded from Q1 because it is a user-level dietary preference / restriction. Breakfast is excluded from Q1 because meal timing belongs to quiz / Plan-level parameters, not cuisine craving.

  - Chinese: `chinese_restaurant`, `chinese_noodle_restaurant`, `dim_sum_restaurant`, `cantonese_restaurant`.
  - Thai: `thai_restaurant`.
  - Indian: `indian_restaurant`, `north_indian_restaurant`, `south_indian_restaurant`.

- No Foursquare-to-Google history migration or matching backfill is required because the database tables were cleared before the provider switch. The active migration path should drop / replace FSQ-specific tables, columns, functions, env vars, and provider config rather than preserving them for compatibility, and fresh local / CI resets should not replay obsolete FSQ / MapKit schema first.
- `member_fetches` / `options`-style active Room storage must not become a Google display-content cache; it should carry only provider identity and app-owned user / vote / verdict state.
- Q5 card / vote telemetry may include Google Place ID when it is app-owned interaction state tied to room / member / card / vote records. This connects a member's rating to the candidate they saw so Q5 inference and verdict ranking can work, even though verdict uses a separate group-level fetch plan. Q5 card display order uses a deterministic app-owned shuffle keyed to the member and `q5_card_set_id`: reopening the same set shows the same order, while a new card set gets a new seed. Google result order must not determine card order. If a Q5 card fails current refetch before the member sees the Q5 set, GetToIt refills the set with a same-axis replacement and shuffles the final three cards once; if same-axis refill cannot be found, it regenerates the set or takes the Q5 no-results path. If a failure happens after any card is visible, GetToIt replaces only the failed slot with a same-axis card, preserves already-seen card positions, and records an app-owned operational receipt; if no same-axis replacement is possible, that member's Q5 attempt fails and they may retry once with a fresh card set under the same locked Room parameters. Retry preserves the same three axis roles and member constraints, but creates a fresh `q5_card_set_id`, fresh deterministic shuffle seed, and fresh candidate Google Place IDs. If the retry also fails, that member's Q5 remains failed and the app stops retrying. The member is not silently dropped; if they exit or the initiator manually closes before retry completion, existing non-exited / submitted-member verdict rules apply. v0.1.0 inference does not correct for card position bias; it persists position / display order so future scoring work can measure bias and add correction with real data. After verdict / history, retain only the member-scoped Q5 learning telemetry, not the full rendered Q5 card set. Durable Q5 telemetry may keep Place ID, card position / display order, card axis role, rating / outcome label, `q5_card_set_id`, `q5_generator_version`, `constraint_profile_version`, and app-authored, place-name-free selection reason codes such as `selected_cuisine_keep_feasible`, `selected_contrast_pool`, `selected_vibe_band_edge`, or `selected_crowd_quality_floor`. It must not include place names, ratings, summaries, addresses, hours, a rendered-card snapshot, raw candidate payloads, provider facts inside reason text, or other Google display fields.
- If a shown Q5 card receives no rating because the member exits or leaves the quiz incomplete, retain a member-scoped non-response outcome label rather than a rendered-card snapshot. The durable record may keep Place ID, card position, axis role, generator / constraint version, and controlled reason code such as an exit or incomplete reason. It must not include Google display content or raw candidate payloads.
- Candidate eligibility receipts may persist Google Place ID plus app-owned keep / cut reason codes when tied to an active Room, Q5 card set, or verdict decision. They must not persist provider values behind the reasons, place names, addresses, ratings, hours, price, distance, summaries, or raw payloads. Broad candidate-pool receipts are active-Room audit state, not a durable place archive; only app-owned verdict receipts survive into Plan history.
- The full scored verdict candidate pool is transient and must not be persisted as a historical archive of every scored Place ID, rank, and score. Durable verdict state keeps only the top-four slate plus lean app-owned receipts needed for audit / learning.
- Verdict persistence may include the ranked verdict slate: up to four Google Place IDs with rank, final aggregate fit score, scoring version, room / verdict IDs, winner / reroll metadata, decided timestamp, and app-owned reason / receipt metadata. The fit score means "how well this Place ID fit this Room's user-entered criteria and Q5 signals under this GetToIt scoring version"; it is not an objective venue quality score. In v0.1.0, it is stored for audit / learning but not shown in the UI. Ranking sorts by final aggregate score first. Only exact normalized score ties use the deterministic app-owned seed in v0.1.0, unless scoring-engine tests prove a tiny versioned epsilon is needed to prevent floating-point instability. The seed is fixed for one verdict generation, so rerolls advance through the existing slate order; new Room / new decision creates a new seed. The tie-breaker must not use Google result order, chunk order, duplicate appearance, provider ranking, or distance inside the committed Search area. Verdict persistence must not include display name, address, Google Maps URI, summaries, raw candidate payloads, provider facts behind the ranking, or component scores that reconstruct provider facts, such as rating score, price score, hours score, distance score, atmosphere score, or summary-derived score. Those provider fields are fetched only for immediate display or used transiently during scoring.
- The Google provider migration decides scoring contract, not exact scoring math. Hard constraints, provider-field eligibility rules, persistence boundaries, verdict slate / reroll behavior, and tie-breaking rules are locked here. Exact formula design is deferred to the scoring-engine / embeddings work so it can be tested against realistic rooms.
- Outcome labels may be persisted as app-owned learning signals tied to Google Place ID, Room / verdict / member context where needed, generator / scoring version, and app-owned reason codes. Examples include voted, accepted, rerolled, user-skipped, abandoned, and reason-selected. Q5 votes and card reactions are member-scoped. Verdict acceptance, reroll, user-skip, abandoned, and reason-selected events are group / verdict-scoped, with `actor_member_id` when one member takes the action. System-caused skips, such as failed current refetch or slate-entry unavailability, are operational receipts rather than outcome labels. These labels record how users reacted to GetToIt's own card or verdict; they must not be stored with Google names, provider facts, summaries, or raw candidate payloads. Future analysis can refetch current Google data by Place ID when policy allows and the analysis needs provider context.
- Durable reason storage in v0.1.0 uses app-authored controlled reason codes only. Reason codes must be place-name-free, provider-fact-free, stable enough for analytics, and interpreted with their event type, Room / verdict / card context, and generator / scoring version. Released reason codes are append-only; if meaning changes, add a new code or version rather than renaming or redefining the old code. The durable categories are Q5 card selection, candidate keep / cut receipts, verdict outcome labels, reroll / user-skip reasons, and operational refetch / failure receipts. Exact enum values are deferred until implementation so they match real event shapes. User-authored free-text reasons are out of scope because they are harder to analyze, carry privacy risk, and could accidentally persist Google display content or provider facts.
- Google `reviewSummary` and `generativeSummary` are requested in v0.1.0 even though they are not displayed. They remain transient provider inputs; embeddings or other derived artifacts from them are not durable until a future policy-aware engine decision explicitly allows it.
- Dietary hard needs, allergies, and Cuisine NEVERs must not be inferred from Google `reviewSummary`, `generativeSummary`, raw reviews, atmosphere text, or future summary embeddings in v0.1.0. Enforce them only from user-owned constraints and high-confidence provider fields / types. For group verdicts, hard allergies, dietary restrictions, and Cuisine NEVERs apply as a union across non-exited members: every candidate must satisfy every active participant's hard safety constraints or be disqualified. Soft dietary preferences are out of scope for v0.1.0 unless they are explicitly modeled as hard constraints. Text-derived dietary safety belongs in a future scoring-engine / embeddings grill because false confidence is a high-cost failure.
- Missing Google atmosphere fields or summaries are not hard eliminators in v0.1.0. They should hurt vibe scoring confidence. Missing price, hours, rating, or user-rating count disqualifies because those fields protect basic recommendation trust.
- The crowd approval quality floor applies both before Q5 card generation and before final verdict ranking, because the winner comes from the final verdict candidate pool rather than necessarily from the three Q5 cards. Rating and user-rating-count thresholds are platform-owned quality standards, not member-selected parameters. The group verdict therefore uses the shared v0.1.0 floor rather than a strictest-member or averaged crowd-approval preference. This ADR locks the crowd approval inputs, minimum floor, and directional preference only; it does not lock the scoring formula. The exact formula should be revisited in the future scoring-engine / embeddings grill because it will interact with Q5 inference, summary embeddings, and group aggregation.
- No-results recovery relaxes qualitative constraints before ending the current attempt. The first relaxation is vibe-band widening on the member-selected 1-5 scale; for example, a selected 4 may expand to 3-5 while preserving preference for candidates near 4. If that is still too thin, crowd approval may relax only while preserving the platform-owned v0.1.0 quality floor: `rating >= 3.7` and `userRatingCount >= 15`. The floor is not a binary scoring target; above it, the engine should still prefer stronger combined rating / count evidence. Price cap, meal-time eligibility, required provider metadata, and the selected Q1 cuisine mappings stay fixed for the current Room. Changing them requires starting a new decision / Room. Cuisine still follows the already-accepted keep / contrast design rather than a sealed-box filter.
- Logs, telemetry, and analytics are treated as durable storage for this policy boundary. Durable app tables may store Google Place ID where it is required app-owned state, such as Q5 votes, candidate receipts, or verdict receipts. Operational logs should prefer app-owned room / candidate IDs, request status / error codes, latency, field-mask / SKU labels, aggregate counts, and reason-code summaries. Google Place IDs may appear only sparingly in structured, access-controlled server logs when needed for debugging, with short retention. Analytics and event streams should not carry Google Place IDs unless they are explicit app-owned interaction records that need the Place ID. Logs, telemetry, and analytics must not include place names, summaries, addresses, ratings, hours, photos, raw response bodies, or other Google display content. Overfetch telemetry is allowed only as aggregate, non-place telemetry: committed radius, provider request radius, overfetch delta, candidate counts before / after trim, and aggregate trim reason counts. It must not include Google Place IDs or other per-place identifiers because trim analysis does not need them.
- v0.1.0 has no durable server-side Places response cache. Per-request / in-memory dedupe is allowed while resolving one user flow, including dedupe between Q5 probe candidates and the final verdict fetch inside the same decision, but the old Foursquare `places` cache table pattern is not carried forward into the Google migration.
- Nearby Search (New) is the primary candidate-pool fetch endpoint for v0.1.0. The existing GetToIt Search area supplies the single app-owned eligibility circle for the Room; it is set by the initiator before launch, shared by all members, and immutable once the Room is created. No new location-selection surface is introduced for the Google switch, and members do not contribute separate locations or radii in v0.1.0. Distance is a hard binary eligibility rule: candidates inside the committed Search area may be considered, candidates outside it are cut, and distance inside the committed circle must not influence ranking. The Google provider layer may overfetch slightly beyond the committed Search area radius to offset provider ranking / edge effects, but it must trim candidates server-side back to the committed Search area before Q5 card generation or verdict ranking. The v0.1.0 provider request radius is proportional with a cap: committed radius plus `min(radius * 0.15, 0.5 mi / 805 m)`. This overfetch radius is not shown to users; UI surfaces show only the committed Search area, because no user-visible candidate may survive outside that circle. Text Search (New) is reserved for later explicit text intent, such as a typed cuisine / place query, not the structured pool builder.
- All Google Places calls go through a Supabase Edge Function. Mobile and web clients do not call Google Places directly. The Edge Function owns API key custody, field masks, response shaping, logging redaction, rate limiting, cost controls, and future provider-change insulation.
- The Google provider layer owns a server-side per-Room call budget for candidate fetches and refetches. The exact numeric budget is implementation-tuned, but the product rule is fixed: deterministic fetch planning, bounded retries, no client-controlled widening of field masks or call counts, a merged final pool small enough that every eligible candidate is scored in v0.1.0, and a no-results path when the budget cannot produce enough valid candidates after allowed qualitative relaxation.
- If Google quota, the per-Room call budget, or cost guardrails are hit mid-Room, GetToIt fails closed into no-results / temporary-unavailable handling. It must not widen constraints, reduce field masks client-side, skip attribution, or fall back to another provider. Durable telemetry is an app-owned operational receipt only: Room / Plan ID, failure type (`quota_exhausted`, `budget_exhausted`, `cost_guardrail_blocked`), timestamp, and step (`q5_probe`, `verdict_fetch`, `refetch`), with no Google payload, display content, stale place names, or provider facts.
- Google field masks are strict server-owned allowlists per use case; clients cannot choose or widen them. v0.1.0 keeps stable named / versioned masks for Q5 display, verdict display, refetch / history, and internal summary validation. Each mask requests only the fields needed for that surface or internal step because field masks control both cost tier and policy exposure. The mask version may be persisted only as app-owned operational metadata when useful for debugging, cost analysis, or behavior reproduction; it does not make returned Google fields durable.
- The Edge Function returns only immediate UI payload plus provider key in v0.1.0: Google Place ID, display name for current rendering, optional formatted address and current Google Maps URI for verdict, and required attribution payload. The Maps URI is fetched or derived for the current response and must not be persisted in verdict / history. It does not return summaries to clients in v0.1.0 if they are not displayed; any server-side summary validation discards summaries before responding.
- Fetching and deciding stay separate. A Google provider Edge Function fetches, shapes, redacts, and enforces policy; a separate GetToIt recommendation / verdict module selects Q5 cards and winners. Clients render assigned Q5 cards and verdict output rather than ranking locally.
- History becomes an app-owned decision record plus refetched provider display, not a durable snapshot of third-party place content.

**References**

- Google Places API policies: Place IDs are exempt from caching restrictions and may be stored indefinitely.
- Google Places API policies: displayed Places API content requires Google attribution; place details, photos, and reviews require returned attributions where provided.
- Google Places API field tiers: `generativeSummary`, `reviewSummary`, and structured atmosphere fields are Enterprise + Atmosphere fields.
- Google Places API billing: field masks determine the highest SKU charged for a request.
- Google Nearby Search (New): `maxResultCount` is between 1 and 20, and type restrictions have provider-defined limits, so a "one fetch" verdict plan may need deterministic server-side chunking if the final request shape exceeds limits. Chunked results are merged by Google Place ID into one transient pool before app eligibility and scoring; duplicate appearances do not boost the candidate, and provider/chunk ordering is ignored for ranking.
