# Workflow overhaul â€” Web invitee single-link flow

Outcome of the `/grill-with-docs` session on 2026-05-21 against [[../15_issues/0.1.0/issues/sg-wf-5-web-invitee-flow|sg-WF-5]] (#158). Sibling to [[0.1.0-workflow-overhaul-plan-setup|0.1.0-workflow-overhaul-plan-setup]] and [[0.1.0-workflow-overhaul-plan-list|0.1.0-workflow-overhaul-plan-list]]. The parent grill ([[0.1.0-workflow-overhaul-plan-setup|0.1.0-workflow-overhaul-plan-setup]] Â§Q6) locked the *what* â€” the Web invitee is the Account-disjoint Plan-member subtype that reaches a single Plan through the iMessage/SMS deep-link, with no homepage and no Plan list. This grill resolved the *how*: the web surfaces, the identity-persistence mechanism, the vote contract, and the resume / read-only / leave behaviors.

## Headline framing

The web fallback today still renders the **retired typed-column quiz**. `web/lib/quiz.ts` writes the typed vote columns (`q1_vetoes` / `q2_budget` / `q3_walk_minutes` / `q4_vibe` / `q5_regret`) and the screen renders the deleted `DUMMY_CANDIDATES` fixture. iOS moved to the generic-jsonb votes schema ([[../60_engineering/adr/0010-generic-jsonb-votes-schema|ADR 0010]]) and the scenario-question / Q5-preference-probe redesign; web silently rotted a whole quiz generation behind.

So the web invitee flow is two pieces of work, not one:

1. The **shell** â€” landing, name entry, resume-on-reclick, read-only verdict, leave. This is what sg-WF-5 now owns.
2. The **quiz port** â€” bringing the web quiz to redesign parity (scenario questions, real per-member Foursquare fetch, Q5 factorial probe). This is a separate sibling tracer-bullet, **tb-WF-10**.

The shell is hollow without the quiz port â€” they are a **delivery pair**. But they are separable because the shell's contract with the quiz is narrow (a vote envelope and a progress envelope), and that contract is exactly what Q2 pins down.

## Grilled decisions (Q1â€“Q8)

### Q1 â€” Scope of sg-WF-5
**Locked:** split the shell from the quiz port.

sg-WF-5 owns the web invitee **shell**: the `/join/<roomId>` landing, name entry, resume-from-state on re-click, the read-only verdict card, and leave. Bringing the web quiz itself to redesign parity is a **separate sibling tracer-bullet, tb-WF-10** â€” it is a vertical of its own (scenario questions, the real per-member candidate fetch, the Q5 factorial probe), not a sub-step of the shell.

sg-WF-5 and tb-WF-10 are a **delivery pair** â€” neither ships a usable web invitee experience alone. They are still filed separately because the seam between them (Q2's vote contract) is clean: the shell hands the quiz a member identity and a room, and the quiz hands back vote slots in a fixed envelope.

**Why split:** the quiz port is large, self-contained, and AFK-able on its own; folding it into sg-WF-5 would make a single issue too big for one clean AFK slice and would couple two unrelated review surfaces (React shell vs. quiz-engine wiring).

**Rejected:** one mega-issue covering shell + quiz (too large for one AFK slice â€” see [[#Then â€” decomposition]]); shipping the shell against the rotted typed-column quiz (would persist typed vote columns the redesigned verdict engine can no longer read).

### Q2 â€” The vote contract
**Locked:** extract a shared leaf module, `supabase/functions/_shared/votes-wire.ts`.

Today the vote wire shape is defined **three times**: in iOS Swift, in `web/lib/quiz.ts`, and in `supabase/functions/_shared/votes-schema.ts`. The web copy is a hand-mirror, and it is the copy that rotted a whole quiz generation behind.

The fix: extract the wire types plus `buildVotesSlotsFromLegacyAnswers` into a new **leaf module** `supabase/functions/_shared/votes-wire.ts` â€” leaf because it pulls in no engine code, only the tiny `Q5Rating` / `Axis` types. The web app **imports it directly**; `web/lib/quiz.ts`'s hand-mirror is deleted.


**Why a shared module over a fourth hand-mirror:** three hand-copies already produced one silent divergence (the web quiz). A fourth copy guarantees the next one. A single imported module makes the contract un-forkable.


### Q3 â€” Identity persistence
**Locked:** the anonymous Supabase session is the sole identity mechanism.

The SMS link `/join/<roomId>` is **identical for every invitee** â€” it is shared into a group chat, so it cannot carry a per-member token. Identity must therefore be **browser-held**. The flow keeps the status-quo anonymous Supabase session (minted via `signInAnonymously()`, persisted in `localStorage`). No URL-embedded member token, no separate cookie.

A direct consequence, accepted and documented: **cross-browser and cleared-storage resume is impossible by construction.** A web invitee who switches devices, switches browsers, or clears storage is a *new* member on re-click â€” their prior vote strands in the room. This is not a bug to be fixed later; it is the cost of a shareable single link with no per-member token.

**Why the anon session over a URL token:** a per-member token in a link that is *designed to be forwarded* is a capability-URL footgun â€” anyone the link is forwarded to inherits the first clicker's identity. The shared link cannot carry per-member state safely.

**Rejected:** URL-embedded member token (capability-URL footgun on a forward-by-design link); a separate first-party cookie (redundant with the `localStorage` anon session, and no better at cross-browser resume); a server-issued claim code on first landing (solves nothing Q3 needs â€” the device still has to hold *something*, and that something is already the anon session). The cross-context bridge problem this raises *for app install* is real, but it is Q8's territory â†’ sg-WF-7.

### Q4 â€” Name entry surface
**Locked:** name-entry-alone landing. No plan summary.

The first click lands on a **name-entry-only** surface â€” a single text input and one CTA. No "[Initiator] wants to figure out dinner" plan summary, because rendering the initiator's name would need a new public RLS read path, and the initiator display-name source is itself the unresolved 0.1.0 #2b blocker.

**Finding (verified during the grill):** there is **no `members.display_name` column anywhere** â€” all 26 migrations were checked. `compute-verdict` currently synthesizes a placeholder name, `"m" + userId.slice(0, 4)`. So this feature introduces the **system's first real display-name source**.

In scope for the shell-wiring tracer-bullet:

- Add a **nullable** `members.display_name` column (additive migration).
- `compute-verdict`'s `fetchVotes` joins it, and keeps the existing `m<uuid>` placeholder as the fallback when the column is null.
- **iOS members still render the `m<uuid>` placeholder** â€” iOS has no name-entry surface. This is flagged as an **adjacency**, not folded into this scope. Closing it is a separate decision (an iOS profile display-name source).

Name-entry UX micro-decisions (locked):

- Single text input, one CTA.
- 30-character cap.
- Placeholder copy: `Your name`.
- CTA disabled until the trimmed value is non-empty.
- Duplicate names across members are allowed (the `user_id` is the identity; the name is a label).
- The name is **not editable after entry** â€” no `members` UPDATE policy is added.

**Why name-entry-alone:** a plan summary would force a new public RLS read path and depends on the unresolved #2b initiator-display-name blocker. The landing's job is to get the invitee a member identity; the plan summary is a nice-to-have that drags a blocker into scope.

**Rejected:** name + plan summary on the landing (needs a new public RLS read path + the #2b blocker); auto-generated name with no entry (defeats the entire point of having a name); editable name after entry (needs a `members` UPDATE policy for marginal value â€” a mistyped name is re-fixable by leaving and rejoining per Q7).

### Q5 â€” Resume behavior
**Locked:** server-side resume via `members.quiz_progress`.

Re-clicking the link mid-quiz **resumes the invitee at their last-answered question**, with prior answers preserved. Restart-at-Q1 was rejected â€” it contradicts the workflow-overhaul Q11 clarification, where the iOS Joined-card resume was explicitly built to *mirror* the web invitee re-click behavior. The two joiner subtypes must stay consistent.

Mechanism â€” and the headline here is **zero new server code**:

- The web app writes `members.quiz_progress` through the existing `members_progress_upsert` RPC (already `grant execute to authenticated`; anonymous sessions qualify; the RPC body pins to `auth.uid()`).
- On boot, the web app reads `quiz_progress` with a plain select.
- The column, the RPC, and the progress contract all already exist â€” landed by [[../15_issues/0.1.0/issues/tb-wf-7-plan-list-joiner-resume|tb-WF-7]].
- The `answers` slots inside `quiz_progress` use the same `{ meta, answer }` envelope as the votes, so resume rides the **Q2 `votes-wire.ts` contract** â€” no separate progress wire type.

The other two re-click cases already work in `SessionRoom.boot`: "already voted â†’ Waiting" and "verdict exists â†’ Verdict."

**Inherited limitation (not re-grilled here):** a resume *into* Q5 re-fires the per-member candidate fetch. This is the same limitation tb-WF-7 documented for iOS; it belongs to the quiz port, **tb-WF-10**, not the shell.

**Why server-side over client-only:** `localStorage` already holds the anon session, but quiz answers in `localStorage` would not survive the Q3-accepted device switch *for the same member* â€” and more importantly, the server already has the progress column and RPC. Reusing them is strictly less code than a client-side mirror.

**Rejected:** restart at Q1 on every re-click (contradicts the workflow-overhaul Q11 mirror lock; brutal under 0.1.0's no-timer sessions that can sit dormant for hours); client-only `localStorage` progress (re-implements what `members.quiz_progress` already does server-side); a new progress-specific RPC or table (the existing `members_progress_upsert` + column already fit).

### Q6 â€” Decided re-click
**Locked:** read-only verdict card, with a terminal fallback.

A web invitee **never has a reroll affordance** â€” reroll is initiator-only ([[0.1.0-workflow-overhaul-plan-setup|0.1.0-workflow-overhaul-plan-setup]] Â§Q9). So `decided-active` and `decided-expired` look **identical** to a web invitee. On re-click of a decided Plan:

- **Membership resolves** â†’ show a **read-only verdict card**: the Plan name and the verdict venue only. No receipts, no per-axis cuts â€” `votes` is ephemeral and is gone by the time a Plan is decided.
- **Membership does not resolve** â†’ fall back to a **"this plan is closed" terminal**. This happens when the member row was purged by the 30-day anonymous-user TTL ([[../60_engineering/adr/0006-privacy-posture-0.1.0|ADR 0006]]), or when a stranger opens a forwarded link.

The read path reuses the existing `plans_decided_for_user` / `plans_history_for_user` RPCs â€” a web invitee returns with `role = 'joined'`, exactly like an Account-member joiner. During `decided-active` the card live-updates on the existing Realtime rebroadcast.

`verdicts` + `options` survive for the read because the room creator is always a **non-anonymous Account member**, and the anon-TTL sweeper only purges `is_anonymous` users. The creator's rows keep the Plan and its verdict alive even after every web invitee's anon row has aged out.

**Why one card for both decided states:** the difference between `decided-active` and `decided-expired` is *the reroll window*, and a web invitee has no reroll. Surfacing a state distinction they cannot act on would be noise.

**Rejected:** distinct `decided-active` vs `decided-expired` web surfaces (the distinction is reroll-window state, which is meaningless to a non-rerolling web invitee); a full verdict screen with receipts / cuts (`votes` is ephemeral â€” the data is gone); always show the terminal "closed" screen (throws away a perfectly good read path for invitees whose membership still resolves).

### Q7 â€” Leave
**Locked:** all three facets ratified.

1. **Affordance placement.** A `Leave` affordance appears on the **Q1â€“Q5 quiz chrome only** â€” not on Waiting and not on Verdict. A web invitee has no Plan list to clean up, so once they are out of the quiz, closing the browser tab is a sufficient exit; a dedicated Leave control there would be redundant chrome.
3. **Soft rejoin.** Re-clicking the link after leaving is a **fresh first-landing** â€” name entry again. No tombstone, no hard block. Quorum self-corrects because the `members` row is gone.

**Why quiz-chrome-only:** the Web invitee's only persistent handle on a Plan is the link itself; once they reach Waiting or Verdict they have effectively committed their vote, and "leave" past that point is just "close the tab." The leave affordance earns its place only while there is an in-progress quiz whose answers a leave would discard.

**Why soft rejoin:** a hard block would require a tombstone row and a block-check on every landing, for no real benefit â€” a web invitee who leaves and comes back is indistinguishable from a new invitee, and treating them as one keeps the landing logic single-path.


### Q8 â€” Second-plan and the app-installed branch
**Locked:** second-plan falls out of Q3; the app-installed gap is closed as its own issue.

**Second plan.** A web invitee invited to a second Plan by a different person is handled *by construction* â€” Q3 gives them one anonymous user, and `members` / `votes` / `quiz_progress` rows are per-room. Two Plans are two independent row sets. This needs only a clarity note in the spec: **name entry runs per Plan** â€” there is no cross-plan name pre-fill, because there is no cross-plan surface to pre-fill from.

**App-installed mid-flow gap.** This one is real. A web invitee who votes in the browser and *then* installs the iOS app and signs in with Apple gets a **fresh Apple `user_id`**, unlinked to their web anonymous `user_id`. Consequences: the in-flight Plan does not appear on their new Plan list, and re-entering it would force a re-vote â€” their browser vote is stranded.

The user directed: **close this gap as its own buildable issue â†’ sg-WF-7 (HITL).** It is not in sg-WF-5's scope and is not resolved by this grill â€” it needs a grill of its own.

**Why second-plan needs no mechanism:** the per-room row model already isolates Plans. There is nothing to build; there is only something to *write down* so a future reader does not invent a cross-plan session.

**Rejected:** cross-plan name pre-fill for a returning web invitee (there is no homepage / list to pre-fill from â€” see [[0.1.0-workflow-overhaul-plan-setup|0.1.0-workflow-overhaul-plan-setup]] Â§Q6 "no Plan list"); folding the app-installed bridge into sg-WF-5 (it is a cross-context identity problem with a real design space â€” see sg-WF-7 â€” not a shell surface).

## CONTEXT.md correction (made during this grill)

The grill surfaced a wrong line in [[../../CONTEXT|CONTEXT.md]] â†’ Relationships. It had claimed "A reroll mints a new Room against the same Plan." That is false. Evidence found while tracing the read path for Q6:

- `apply_reroll(p_room_id)` mutates the **same** room in place.
- `rerolls.room_id` + the `tg_rerolls_cap` trigger enforce a **per-Room** 3-burn cap.
- tb-WF-8's `tg_rerolls_maybe_expire_plan` counts `rerolls WHERE room_id = X` and expects that count to reach 3 â€” which only holds if every reroll stays on one room.

The line is now corrected to: "A reroll re-runs the verdict on that **same** Room in place â€” it does not mint a new Room (the 3-burn cap is per-Room)." Recorded here because the correction is a grill finding, not a content edit owned by any issue.
