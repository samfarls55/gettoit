## Headline framing

A [[../../CONTEXT|Web invitee]] votes in the browser, then installs the
iOS app and signs in with Apple. The browser and the app keep identity
in separate storage contexts that cannot see each other: the browser
holds a throwaway **Anonymous session**; Apple sign-in mints a
**separate, brand-new account**. The browser vote is orphaned, and
re-entering the Plan in the app forces a re-vote.

The fix is **transport, not new linking machinery**. The destination
is already built: `RootView.shouldRenderSignInGate` fires on
`.anonymous` as well as `.idle`, and S00a's `SignInScreen` already
routes the Apple tap through `linkApple` (preserving `user_id`, zero
row migration) whenever the keychain holds an anonymous session. So
sg-WF-7's whole job is to get the web invitee's anonymous session out
of the browser and into the app keychain â€” **before** the Apple tap,
because `linkApple` only upgrades a *currently anonymous* session.

## Grilled decisions (Q1â€“Q6)

### Q1 â€” Bridge scope: same-device
**Locked:** same-device only â€” browser and app on one phone.

The web invitee reaches the Plan through an SMS/iMessage link, opens
it in mobile Safari, and installs the app on the same phone. The
anonymous session only ever has to cross from Safari's `localStorage`
to the app keychain *on one device*.

**Why:** mirrors the web-invitee grill's Q3 ("cross-device /
cleared-storage resume is impossible by construction â€” the cost of a
shareable single link"). Cross-device claim is a strictly harder
problem the product does not promise.

**Rejected:** a cross-device bridge (laptop-web â†’ phone-app) â€” out of
scope by the same reasoning as Q3.

### Q2 â€” The bridge channel: claim code
**Locked:** a claim code, entered on S00a before the Apple tap.

One technical fact decides the channel: `linkApple` only works on an
anonymous session, so the anonymous session must reach the keychain
**before** Apple sign-in completes. Each candidate is judged against
that ordering constraint first.

- **Clipboard token** and a **third-party deferred-deep-link SDK**
  both *solve* ordering automatically (the context is present before
  S00a renders) â€” but the clipboard is unreliable (the multi-minute
  install gap, the iOS paste banner, staleness) and fails *silently*;
  the SDK is a heavy paid dependency that falls back to the clipboard
  or to privacy-hostile fingerprinting.
- A **Universal Link "Open in app" round-trip** cannot survive an app
  install, so the first launch never carries it; making it work needs
  the *same* "act before Apple sign-in" discipline as a code, plus
  app-switching and a capability token in a URL â€” strictly worse.
- A **claim code entered on S00a** makes the only correct order the
  default order: code entry and the Apple button are the same screen,
  and the code must be entered first for the Apple tap to become
  `linkApple`.

**Why:** for a rare, one-time, high-value conversion event,
structural correctness + reliability beat saving keystrokes. The code
needs no SDK, no clipboard, no capability URL, and its failures are
*visible* and recoverable.

**Rejected:** clipboard token (unreliable, silent failure, paste
banner); deferred-deep-link SDK (heavy paid dependency, fingerprinting
liability); Universal Link round-trip (cannot survive install; same
discipline as a code but worse). See
[[../60_engineering/adr/0015-web-invitee-account-claim-bridge|ADR 0015]].

### Q3 â€” Claim scope: per-person
**Locked:** one code carries the whole anonymous identity.

The anonymous identity *is* a session key; the code carries that key,
not a single Plan. Redeeming one code brings over **every** web Plan
that anonymous identity voted in. The feature is "claim your account,"
not "claim this Plan" â€” consistent with the issue title.

A forced consequence, accepted: there is no Supabase primitive to mint
a session for an arbitrary anonymous user, so the code must carry the
session key (the refresh token) itself. It is stored server-side
encrypted, single-use, short-TTL. Low stakes â€” the protected asset is
an anonymous invitee's vote in one dinner Plan.

**Why:** the key is the person; all their rows are keyed off one
`user_id`. Bringing the identity brings everything, and the user never
has to hunt down every link â€” reaching one room is enough to mint.

**Rejected:** per-Plan claim (mechanically impossible without the
deferred merge â€” the key is the whole identity).

### Q4 â€” Surfacing: low-key web affordance, S00a entry, lazy mint
**Locked:** all three.

- **Web side.** A low-key affordance ("Getting the app? Bring this
  Plan with you") â€” a quiet line, not a banner, not a hard upsell.
  Tapping it **lazily mints** the code and reveals it with plain
  instructions pointing at the app-side entry.
- **App side.** A secondary "Voted on the web?" entry beneath the
  Sign-in-with-Apple button on S00a; tapping it reveals a code field.
  A fresh user who was never on the web simply ignores it.
- **Lazy mint.** The code is generated on the web-affordance tap, not
  eagerly â€” keeps the Waiting screen clean, never mints unused codes,
  and (because the code carries a live session key that can go stale)
  mints it at the moment of intent.

**Why:** the web side *teaches* and the app side *receives*. The
low-key web affordance respects the web-invitee grill's Q7 ("the web
fallback is plumbing, not a growth surface") â€” it helps a user who
already decided to install, rather than pushing anyone.

**Rejected:** an eager always-visible code on Waiting (clutter, waste,
staleness); a prominent app-conversion banner (violates
plumbing-not-growth); an always-visible S00a code field (noise for the
common fresh-user path).

### Q5 â€” Recovery: mint affordance also on the read-only verdict card
**Locked:** the mint affordance lives on every "membership resolved"
web surface â€” Waiting **and** the read-only verdict card. Off the quiz
(finish voting first) and off the terminal "this plan is closed"
screen (nothing to claim).

A returning user reconnecting *old* data lands on **decided** rooms,
which render the read-only verdict card â€” not Waiting. A
Waiting-only affordance would leave that user with nowhere to mint.
Because the claim is per-person (Q3), reaching one old room is enough
to recover everything.

The app-side code-entry surface carries **teaching copy** â€” how to go
generate a code from any prior link if the user does not already have
one. Copy must be **honest about the ~30-day ceiling**: per
[[../60_engineering/adr/0006-privacy-posture-0.1.0|ADR 0006]], anonymous
identities are purged after 30 days; older links land on the terminal
"closed" screen with nothing to recover. Say "bring back your recent
web Plans," not "recover all your history."

**Why:** the recovery scenario (voted in several rooms over time, then
installs the app) is real and worth supporting â€” but it only works if
the affordance is reachable from a decided room.

**Rejected:** Waiting-only affordance (breaks old-room recovery);
mint affordance on the quiz chrome (distracts from voting) or the
terminal closed screen (nothing to claim).

### Q6 â€” Failure boundary: before-sign-in only
**Locked:** the claim is **before-sign-in only**. All after-sign-in
recovery is **deferred to a future feature**.

Handled cleanly already: a bad/expired/mistyped code â†’ a visible
error, retry; a claim that succeeds then an Apple sign-in that fails â†’
the anonymous session is safely in the keychain, retry the Apple tap;
a claim then a force-quit â†’ the keychain keeps the anonymous session,
next launch S00a routes straight to `linkApple`.

The one permanent boundary: **if the user signs in with Apple without
claiming first, that web data is stranded.** Once a Linked-Apple
session exists, bringing the web identity in is a true two-user merge,
which is out of scope (see below). The only recovery is
delete-and-reinstall â€” which returns the user to a fresh S00a where
the before-sign-in claim works again â€” within the 30-day window. This
is not new harm: it is exactly today's ADR 0007 behavior ("web users
can't claim â€” flagged"); the claim feature is purely additive.

**After-sign-in claim â€” deferred.** The grill sized it:

- **Empty Apple account** (just signed in, did nothing else â€” the
  realistic majority case): supportable cleanly by "reset sign-in and
  claim" â€” delete the empty Apple husk, claim, re-run Apple sign-in as
  `linkApple`. T-shirt **S**, pure composition of existing parts
  (`delete-user` + claim redeem + `linkApple`).
- **Populated Apple account** (created/joined Plans before claiming):
  a true two-user merge â€” T-shirt **L**, the only path that can
  *destroy data*. Needs re-keying every user-FK'd table, an unresolved
  conflict policy (the same user as both a web member and an app
  member of one room), live-room hazards, and an irreversible-migration
  test burden.

Decision: ship **before-sign-in only** now; defer **both** after-sign-in
variants to a future feature. Pre-public-launch with zero real users,
an in-app recovery path â€” let alone an irreversible merge engine â€” is
premature.

**Why:** before-sign-in claim captures essentially all real demand
(a web invitee installs the app and claims in one S00a sitting); the
delete-and-reinstall escape hatch covers the rest for free; the
merge's unique coverage ("used the app for weeks, then remembered") is
rare and not worth its risk now.

**Rejected:** an in-app (post-S00a) claim entry point (deferred);
the full populated-account merge (deferred â€” see ADR 0015 Â§Considered
options 4).

## Mechanics (recommended defaults â€” implementation detail)

- **Claim code:** single-use (burned on redeem), ~30-min TTL,
  8 characters from an unambiguous alphabet (no `O/0`, `I/1/l`).
- **Re-mintable:** an expired code is recovered by tapping the web
  affordance again for a fresh one. The 30-min code TTL is distinct
  from the 30-day *identity* TTL.
- **Redeem endpoint:** rate-limited against code-guessing.
- **Storage:** a `claim_codes` table maps the code to the web
  session's refresh token (encrypted at rest, short TTL, single-use);
  two edge functions â€” mint (authed by the live web session) and
  redeem (protected by the unguessable code + rate limiting).

### Known edge cases (doc-notes, not grilled branches)

- **Apple ID already used on this app.** If the installing user is not
  a true first-timer (their Apple ID is already linked to another
  Supabase user), `linkApple` onto the claimed web user fails
  "identity already linked." The user falls back to a normal sign-in;
  the web vote strands.
- **Legacy-carryover anonymous session in the keychain.** A pre-S00a
  install may already hold an anonymous session at S00a. Redeeming a
  claim code replaces it with the web identity, abandoning the legacy
  one. Rare and shrinking population; the user is making that choice.
- **Multiple codes minted.** Tapping the web affordance repeatedly
  yields multiple codes for one identity; single-use redemption means
  the first redeemed wins, the rest expire (or a fresh mint may
  invalidate prior unredeemed codes â€” implementation choice).
