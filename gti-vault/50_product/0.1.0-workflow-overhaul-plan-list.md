# Workflow overhaul — Plan list surface

Outcome of the `/grill-with-docs` session on 2026-05-20 against [[../15_issues/0.1.0/issues/sg-wf-4-plan-list-surface|sg-WF-4]]. Sibling to [[0.1.0-workflow-overhaul-plan-setup|0.1.0-workflow-overhaul-plan-setup]] (Setup screen scope), this doc locks the **Plan list surface** decisions that were explicitly out of scope of the 2026-05-19 grill.

## Headline framing

The Plan list is the new app landing (parent grill Q3). This doc resolves how it looks and behaves: sectioning, card content, status signaling, joiner distinction, ordering, history visibility, the create affordance, the delete affordance, and the Created-vs-Joined tap-behavior split.

## Grilled decisions (Q1–Q8)

### Q1 — List structure
**Locked:** sectioned by status. Three sections: `Pending` → `Decided` → `History`. Each section renders nothing when empty. The all-three-empty state is the locked empty-state hero CTA (parent Q3).

History section is collapsible (iOS-native disclosure pattern). Default state: expanded on first viewing, sticky-collapsed thereafter per user preference within the session.

**Why sectioned:** the parent grill anchored on the Reminders metaphor (parent doc line 36, "Reminders-app spirit"); Reminders is sectioned, not a flat feed. Sun-yellow being the only state color (`design-system/AGENTS.md` Do/Don't — no red, no green) means status badges have limited semantic palette; section headers carry the meaning much more cleanly than per-card badges. Solves three open issue items in one stroke (status badges, ordering, history visibility) — see Q3/Q7 below for the derived simplifications.

**Rejected:** single chronological list (forces per-card status badges on every row in a no-red/no-green system; can't host History collapse cleanly; works against the Reminders metaphor); grouped-but-flat (intermediate that loses both the simplicity of (a) and the structure of (b)).

### Q2 — Card content
**Locked:** Plan name on every card; **verdict place name as a secondary line on decided cards only**. Pending cards = 1-line (name). Decided / History cards = 2-line (name primary + verdict place name secondary). No thumbnail image at 0.1.0.

Single-line truncation with ellipsis for long Foursquare names (e.g., `"Joe's Stone Crab Restaurant & Patio"` → `"Joe's Stone Crab Restaurant & P…"`).

**Why:** the verdict is the reason the Plan exists; hiding it behind a tap defeats the at-a-glance utility a list provides. Specifically: a History row that reads only the Plan name is useless (the user already remembers they had Thursday dinner; what they want from the list is "what did we end up doing"). The 1-line vs 2-line asymmetry between sections *is signal* — Pending feels unfinished, Decided/History feels concluded.

**Rejected:** name-only across all sections (defeats at-a-glance utility, History becomes a tap-each-row-to-remember chore); name + thumbnail image (expensive MapKit / Foursquare photo fetch on every list render; defer as polish if testing later shows real signal that users want it).

### Q3 — Created vs Joined affordance
**Locked:** small `JOINED` eyebrow chip in `var(--sun)`, top-leading of Joined cards above the name. Created cards carry no chip (Created is the default — mark the exception, not the default).

Chip uses existing C-11 Eyebrow typography (Inter 700 / 11 / tracking 0.18em UPPERCASE). NOT `Joined`, NOT `Invited`, NOT `From Sam`.

**Why:** marks the exception (joined) rather than the default (created); reuses sun-yellow accent (the only accent available); doesn't disturb the name + verdict stack. Joined cards' three-dot menu offers `Leave plan` instead of `Delete plan` (see Q4) — the chip telegraphs which menu set you'll see.

**Rejected:** avatar dot (premature identity rendering; 0.1.0 has no member-identity-on-list affordance anywhere else); sub-subsections inside each status section (heavy nesting on a low-volume surface); no card-level distinction (breaks at-a-glance utility, especially for the "find the plan Sam invited me to" scenario).

### Q4 — Delete / Leave affordance
**Locked:** three-dot trailing menu (`⋯`) on every owned card. Menu contents vary by section + role:

| Card | Menu contents |
|---|---|
| Created Pending | `Edit plan` / `Delete plan` |
| Created Decided | `Delete plan` |
| Created History | `Delete plan` |
| Joined (any status) | `Leave plan` |

Destructive confirms via C-16-pattern bottom sheet. **No red anywhere** — Sunset Pop forbids red/green as state colors; the destructive weight is carried by copy + the warning treatment of the sheet, not by a red button.

**Confirm sheet copy** (context-sensitive):

| Section | Title | Body | Primary pill | Dismiss |
|---|---|---|---|---|
| Pending | `Delete this plan?` | `Nothing's been decided yet — no one's been notified.` | `Delete plan` | `KEEP` (eyebrow) |
| Decided-active | `Delete this plan?` | `The active room will end. Joiners will see a session-ended notice.` | `Delete plan` | `KEEP` |
| Decided-expired | `Remove from history?` | `The verdict will be deleted permanently.` | `Remove` | `KEEP` |
| Joined (Leave) | `Leave this plan?` | `Your answers will be removed. The room continues for everyone else.` | `Leave plan` | `STAY` |

Sheet uses C-16 visual primitive — radius 18, glass background, scrim — same as existing reroll bottom sheet.

**Why three-dot vs swipe-to-delete:** iOS-native swipe-to-delete expects a red trailing reveal; we cannot supply one (no red token). Three-dot is a fully-owned UI element we can style in the Sunset Pop language with no exception. Also: `Edit plan` becomes a discoverable menu item for Pending cards (the tap-pending → Setup-Edit shortcut from parent Q11 is otherwise invisible — the menu gives it a verb).

**Rejected:** native swipe-to-delete (forces a red exception or sun-yellow-as-destructive — semantically wrong); long-press → iOS system context menu (can't fully restyle the system-provided menu); edit-mode toggle in nav chrome (heavier ergonomics, less discoverable than always-visible `⋯`); delete inside Setup-Edit only (no path to delete History rows whose tap goes to read-only verdict, not Setup).

**Friction is appropriate.** Plan delete kills the active room and punts joiners (`Plan delete` in [[../../CONTEXT|CONTEXT.md]]). Three taps (open menu → tap Delete → confirm sheet) is the right ceremony for that weight.

**Implementation flag:** new component primitive needed — the three-dot popover menu. Provisional spec name: **C-25 Action Dot Menu**. Reuse opportunity: same primitive can host the Verdict screen's "View on Maps / Share / Reroll" overflow later.

**Open sub-decision (deferred):** whether to also wire a *secondary* swipe-to-delete with a sun-yellow desaturated reveal as a muscle-memory fallback. Default: no — keep one-affordance for 0.1.0; revisit only if testing shows real complaint about gesture-discovery.

### Q5 — Plan creation: FAB + disambig sheet
**Locked:** bottom-right **floating action button** as the persistent populated-state create affordance. Tap → C-16-style bottom sheet with two stacked ghost C-05 pills: `Solo` (top), `Group` (below). No Cancel button — system swipe-down / tap-scrim dismiss.

The disambig sheet is a new use of the C-16 primitive (sheet) + the C-05 primitive (pill, ghost variant) — no new sheet component. **The FAB itself is a new design-system component.** Provisional spec name: **C-26 Floating Action Button** (Sunset Pop primitive: circular glass button, ~56pt diameter, sits 18pt off the bottom + trailing edges, `var(--sun)` glyph on glass bg, 14pt blur backdrop, light shadow).

**Amends parent doc Q7 (Setup screen inventory):**

- **Solo path** (user picks `Solo` on the disambig sheet) → Setup screen renders **5 controls** — the `Who's coming` row is **omitted entirely**.
- **Group path** (user picks `Group`) → Setup screen renders **6 controls**; the `Who's coming` chips are **`Two of us / A group` only** (the `Just me` option is removed because the user already disambig'd at the sheet).

Parent Q4 (Setup primary CTA copy) is **unchanged**: solo → `Start the quiz`, group → `Drop the invite link`. The disambig just moves the chip-tap one screen earlier.

**Why FAB + disambig vs chrome `+`:** founder preference for the FAB-unfold aesthetic. Cost was a new design-system component (C-26) and an amendment to parent Q7's "flat 6-control layout." Amendment is bounded: the locked one-screen layout is preserved; one row's choice is just pre-determined upstream.

**Why no Cancel:** modal sheet has built-in dismiss (swipe-down, tap-scrim). A visible Cancel button on a 2-option sheet is redundant noise.

**Why disambig instead of just landing on Setup with the 3-chip row visible:** lifts the most-discrete binary (Solo vs Group) up to a moment of dedicated focus. The 3-chip row inside Setup was the parent grill's compromise when the chip lived among 5 other controls; now that it's lifted, it gets the simpler 2-option treatment without `Just me` artificially competing with `Two of us / A group`.

**Rejected:** top-trailing chrome `+` glyph (founder preference was the FAB-unfold pattern); persistent bottom dock pill (occupies vertical real estate the list itself should own; iOS Reminders/Notes don't do this); multi-step wizard (reopens parent Q7's flat-layout decision — explicitly rejected).

### Q6 — Unified entry path
**Locked:** the empty-state hero CTA and the FAB **both route through the disambig sheet**. Single mental model, single code path. The Setup screen's layout (5-control solo / 6-control group) is determined by **one variable** — the disambig choice — regardless of which entry point invoked it.

Empty-state pill: giant C-05 primary pill, mid-screen, full-width. Copy: `Create your first plan` (locked in parent Q3 line 60). Tap → disambig sheet (not Setup directly).

Populated state: FAB in bottom-right per Q5. Tap → disambig sheet.

**Why unified:** keeps Setup's rendering state to exactly two shapes (5-control solo, 6-control group). A third "neither pre-selected, render all 3 chips" rendering — which (b) first-launch streamlining would require — adds a conditional-layout-by-entry-path maintenance smell. First-launch users learn the disambig pattern early; next time they see it, it's already familiar.

**Rejected:** first-launch streamlining — empty-state pill → Setup directly with `Who's coming` as a 3-chip row. Reason: forces Setup to support three render modes; teaches a one-time "tap to Setup directly" flow and then surprises the user on second launch.

### Q7 — Ordering within sections
**Locked:** section-appropriate "newest event first":

| Section | Sort key | Tiebreaker |
|---|---|---|
| Pending | `created_at DESC` | n/a |
| Decided | `verdict_fired_at DESC` | `created_at DESC` |
| History | `expired_at DESC` | `created_at DESC` |

**Why per-section sort key:** each section has its own dominant temporal event. Pending = "you just made it." Decided = "verdict just fired; reroll window is running now." History = "this just became history." Sorting on the section-relevant event aligns each section with the user's mental model of what's happening within it.

**Why not pure chronological by `created_at`:** counterintuitive ordering in Decided / History. Concrete failure: I create Plan A on Monday, B on Tuesday, C on Wednesday. I never run A or B; they sit Pending. I run C immediately and it decides. Under pure `created_at`, my Decided section shows C at the bottom (because A and B were created first) — but C is the only Decided one and just happened. Wrong feel.

**Section transitions surface at top.** When a Plan moves Pending → Decided (verdict fires), it appears at the top of Decided. When a Plan moves Decided → History (window close / third burn / check-in), it appears at the top of History. The user sees state changes immediately rather than having to hunt for them.

**Rejected:** pure `created_at DESC` everywhere (per-section counterintuitive); algorithmic "smart" sorting weighted by actionability (forces user to learn the algorithm; defer until usage signals real need).

### Q8 — Tap behavior on Joined cards
**Locked:** resume-from-state. Tap takes a joiner to wherever the Plan currently is for them.

| State | Tap destination |
|---|---|
| Pending, joiner hasn't opened quiz | S03 Quiz Q1 |
| Pending, joiner mid-quiz | S03 Quiz at last-answered question (per-joiner state) |
| Pending, joiner finished quiz | S04 Waiting |
| Decided-active | S05 Verdict (read-only — joiner can't reroll; initiator-only per parent Q9) |
| Decided-expired | S05 Verdict read-only history |

**Clarifies parent doc Q11.** Parent Q11 said "tap a `pending` Plan → opens Setup in Edit mode." That contract applies to **Created Plans only**. Joined Plans cannot open Setup-Edit (joiner is not the creator). Joined-pending tap behavior is per the table above.

**Why resume-from-state:** already locked elsewhere for Web invitees. [[../../CONTEXT|CONTEXT.md]] → `Plan member`: Web invitees re-opening their SMS link "return to the Plan's current state (quiz / waiting / verdict / read-only history)." The Joined-card tap on iOS is conceptually the same action; using the same semantics keeps both joiner subtypes consistent. Per-joiner quiz progress is already persisted by the 0.1.0 quiz state machine; tap-resume just reads it.

**Rejected:** new "Plan status" read-only view (introduces a surface joiners don't need — they have no admin actions on the room); "tap → quiz Q1 always" (wastes existing partial-answer state; with 0.1.0's no-timer lock, sessions can sit dormant for hours — restarting from Q1 every tap is brutal).

**Implementation flag:** list-render needs to query per-joiner quiz progress for Joined cards. One extra column or one batched query.

**Edge cases:**
- Exited joiner: Plan disappears from their list per `Plan exit` semantics. Tap is never invoked.
- Joiner mid-quiz when initiator deletes: "session ended" toast, Plan disappears. Tap never invoked.
- Decided-active where joiner never finished quiz: still routes to S05 read-only. Joiner sees the result without having voted. Consistent with 0.1.0 quorum-of-one rule.

## S00 Landing retirement

The Plan list **is** the new app entry surface. S00 Landing is fully retired. The implementation slice (separate AFK issue) carries:

- S00 Landing surface doc gets `status: superseded` + `superseded-by:` pointing to the new Plan list surface doc.

## Component delta

Decisions in this grill introduce **two new design-system component primitives**:

| Component | Provisional name | Purpose |
|---|---|---|
| Action Dot Menu | C-25 | The trailing `⋯` glyph + popover menu used on each owned Plan card. Reusable elsewhere (Verdict overflow, etc.). |
| Floating Action Button | C-26 | Bottom-right circular create button. Sunset Pop primitive — glass + sun glyph, ~56pt, light shadow, 18pt off bottom + trailing edges. |

Both are real component-spec additions; surface doc + JSX should land together with `components.md` entries.
