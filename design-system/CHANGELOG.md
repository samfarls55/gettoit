## 2026-05-26

- **S09 Settings — surface escape moves to a top-leading `xmark` close glyph; bottom DONE retired** (`surfaces/09-settings.md` §"Components used" + new §"Surface escape" + amended §"CTA dock hierarchy" + §"Copy register" Close entry + §"Behavior", `code/screens/ScreenSettings.jsx`, `ios/Sources/App/SettingsScreen.swift`, `ios/Tests/SettingsScreenTests.swift`). Workflow-review whole-app audit (2026-05-26) finding #29 flagged the wfr-07 hierarchy as breaking iOS habituation (P-07): a center-aligned bottom-dock `DONE` text label is not where iOS users look to dismiss a modal-utility surface. New treatment: a top-leading `xmark` SF Symbol icon-button (44pt tap target, white-at-0.86 opacity) anchored above the eyebrow on the safe-area inset matches the iOS sheet-dismissal convention. The wfr-07 ghost destructive treatment on `DELETE MY DATA` is preserved verbatim; the CTA dock now holds only that one action. Icon (not text) chosen because Settings is a utility surface that steps out of the Sunset Pop ritual arc — the glyph reads as "sheet escape" at a glance, distinguishing it from the QuizChrome / VerdictReadOnly text-label chrome verbs that name in-arc consequences. iOS contract: `SettingsScreen.Style` drops `donePillFill` / `donePrimaryOrder` / `deleteSecondaryOrder` (no primary/secondary pair to order anymore) and gains `closeSymbolName` / `closeAlignment` / `closeMinTapTarget` / `rendersBottomDonePill` so a regression cannot silently re-introduce a bottom DONE pill or flip the close glyph to top-trailing. The same `onDone` callback the retired pill used now fires from the close glyph — host wiring in `RootView` is unchanged. (wfr-29, #309.)
- **S06 Hard-close — new `Home` chrome row** (`surfaces/06-hard-close.md` new §"Locked chrome (Home)", `ios/Sources/App/LockedScreen.swift`, `ios/Tests/LockedScreenTests.swift`). The original 0.1.0 spec landed the user on the locked verdict with no path back — a terminal-state trap that violated foundation P-01 *Safe Exploration* and the *Escape Hatch* pattern. New top-leading `Home` text-verb chrome row above every other surface layer (above the veil + shutters + plate), mirroring `surfaces/05-verdict.md` §"Verdict chrome (Home)" and the QuizChrome `Back` slot. Eyebrow-token treatment (Inter 700 / 11 / 0.18em / UPPERCASE / white 0.78), 44pt min hit row, top-trailing slot intentionally empty (the locked verdict has nothing to exit — the session already ended). Pure navigation — tap pops the locked screen back to S00 Plan list via the RootView precedence-chain fallback. No session teardown, no membership mutation: the verdict is sealed against re-litigation (`CONTEXT.md` → *Plan / Room lifecycle*), but the user is not trapped on the surface. iOS port: `LockedScreen` gains an `onHome: @escaping () -> Void = {}` initializer parameter (defaulted to no-op so existing call sites stay source-compatible), a `homeChromeLabel = "Home"` static constant matching `VerdictScreen.homeChromeLabel`, and a `homeChromeRow` subview wired above the plate at `zIndex(6)`. Not prefixed `BREAKING:` — the new parameter is defaulted and the existing call sites in `LockedScreenTests` keep compiling untouched. (wfr-12, workflow-review whole-app audit finding #12.)
- **S09 Settings — CTA dock hierarchy inverted; DELETE demoted to ghost, DONE promoted to white pill** (`surfaces/09-settings.md` §"Components used" + new §"CTA dock hierarchy" + §"Copy register" DONE entry, `code/screens/ScreenSettings.jsx`, `ios/Sources/App/SettingsScreen.swift`, `ios/Tests/SettingsScreenTests.swift`). The original 0.1.0 spec rendered DELETE MY DATA as the C-05 white PillCTA (the only action on the surface) and DONE as a mono-tag footer link. Workflow-review whole-app audit (2026-05-26) finding #7 flagged the inversion: the destructive verb visually dominated the user's exit verb. New hierarchy stacks DONE (C-05 white PillCTA) above DELETE MY DATA (C-05 ghost PillCTA — transparent fill, 1.5pt white-0.5 stroke) in the CTA dock. The `tokens.md §1.3` no-red contract still governs — destructive weight lives in the outline + the literal copy ("Delete my data" → alert "Delete forever") + the native two-step confirm alert, never in a colored fill. Sun-fill was not considered (sun = "the system registered your input", never destructive). The two-step confirm alert flow is preserved verbatim; only the trigger's visual register changes. iOS contract pinned via `SettingsScreen.Style` static constants (`donePillFill`, `deletePillFill`, `ghostStrokeWidth`, `usesRedDestructiveColor`, render-order ints, locked alert copy strings) so a regression cannot silently re-promote DELETE without flipping `Style.deletePillFill`. (wfr-07, #248.)

## 2026-05-24

- **BREAKING: S05 Verdict — `cuts` mode + "See what got cut" drawer fully retired** (`surfaces/05-verdict.md` §Modes / §"five-second test" / §"read-only" / §"Edge cases", `components.md` §C-13, `accessibility.md` §"Tap targets" / §"Focus order" (Verdict block renamed `default / committed`) / §"VoiceOver labels" / §"Audit results", `motion.md` §"Easing curves" / §"Mode-specific reveal" / §"Utility motion" / §"Reduced motion fallback", `code/screens/ScreenVerdict.jsx`, `ios/Sources/App/VerdictScreen.swift`). The cuts drawer was the only friction-free change-of-mind path on the verdict — a user could re-pick from the just-eliminated candidates without paying any of the reroll's friction (3-burn cap, stated reason that becomes a new constraint). That is not intended behavior; the reroll path is the canonical re-decide channel, and the cuts drawer was a parallel, friction-free path to the same outcome (load-bearing rationale from the bug-26 grill). Pairs with bug-22 (`Home` verb is pure navigation) to leave reroll as the only re-decide channel. **Fix shape:** the `cuts` mode is removed from the surface doc, the C-13 entry is retired in place (slot left empty — re-numbering would invalidate every existing C-NN citation in the surface / accessibility / port docs), the JSX `showCutsDrawer` flag + `cutsOpen` state + expanded-drawer block are deleted, the iOS `.cuts` enum case + `cutsExpanded` `@State` + `cutsDrawer` view + `showCutsDrawer` / `cutsExpanded` `ModeSnapshot` fields are deleted, and the accessibility + motion specs drop every cuts trigger / row / VO entry. `VerdictScreen.Verdict.cuts` (the value-type data field) is retained — the engine still writes `option_cuts` rows; the surface simply no longer reads them. Any consumer that constructed `mode: .cuts` (Swift) or `mode='cuts'` (JSX) will break — that is intentional. (bug-26, #226.)
