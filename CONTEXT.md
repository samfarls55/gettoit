# CONTEXT

Domain glossary and shared vocabulary for GetToIt. Agents read this before exploring; outputs (issue titles, refactor proposals, test names) should use the terms defined here.

This file is grown lazily by `/grill-with-docs` as terms get resolved during real work. Don't pre-fill speculatively.

## Language

### Auth identity

**Anonymous session**:
A Supabase auth session minted via `signInAnonymously()`, carrying a real `user_id` but no Apple identity. Default for both the legacy Swift app and web prior to the S00a sign-in gate. In the active React Native app under `mobile/`, anonymous sessions only exist as transient legacy state from pre-S00a installs and on the web fallback.
_Avoid_: guest user, anon account, unauthenticated session (a session is still present), no-account user.

**Linked-Apple session**:
A Supabase auth session whose `user_id` has an Apple identity attached, either via fresh `signInWithApple` (no prior session) or `linkApple` (existing anonymous session upgraded, same `user_id` preserved). Canonical mobile post-S00a state.
_Avoid_: signed-in user, authenticated user, Apple account (the account is on Apple's side, the session is ours).

**S00a Sign-in Gate**:
The surface rendered on mobile launch when the current auth state is anything other than Linked-Apple. One Sign-in-with-Apple pill, no skip. 0.1.0 closure of the mobile-app half of [[gti-vault/60_engineering/adr/0007-auth-anonymous-default-apple-upgrade|ADR 0007]]'s anonymous-default. Active implementation belongs in the React Native app under `mobile/`.
_Avoid_: sign-up page, sign-up screen, onboarding screen, welcome screen, account creation page (see Flagged ambiguities).

**Account claim**:
The process by which a Web invitee who installs the mobile app carries their browser-held Anonymous session into the app *before* completing Apple sign-in, so that S00a upgrades that existing identity via `linkApple` (preserving `user_id`, zero row migration) instead of `signInWithApple` minting a fresh, disjoint account. **Same-device only** — browser and app on one phone. **Before-sign-in only** — the claim must complete on S00a *before* the Sign-in-with-Apple tap; once a Linked-Apple session exists there is no in-app recovery (that would require an account merge — deferred to a future feature). If the claim is skipped, Apple sign-in mints a new `user_id` and the web vote strands; the only recovery is delete-and-reinstall (which returns the user to a fresh S00a), within the 30-day anonymous-identity TTL. The carrier is a **Claim code**. Active app work lives in `mobile/`. See [[gti-vault/15_issues/0.1.0/issues/sg-wf-7-web-invitee-account-claim|sg-WF-7]].
_Avoid_: account merge (no merge — the Apple identity attaches to the *existing* anonymous user; nothing is merged), account linking (collides with `linkApple`, which is only the final upgrade step, not the cross-context transport).

**Claim code**:
The short, single-use, short-lived code shown on the web fallback that carries a Web invitee's Anonymous session key from the browser into a freshly-installed app. Entered on S00a *before* the Sign-in-with-Apple tap. **Per-person, not per-Plan** — redeeming one code brings over every web Plan that anonymous identity voted in, because the code carries the whole identity, not a single Plan.
_Avoid_: invite code (the SMS deep-link is the invite; the claim code is a different artifact, used post-install), pairing code, OTP (not an authentication factor — it is an identity-transport token).

### Quiz redesign vocabulary (0.1.0, 2026-05-15)

**Parameter**:
A session-level setting captured BEFORE the quiz begins, on a session-setup surface, typically by the initiator. Distinct from quiz questions (asked of every member) and profile data (account-level, sticky). Parameters are locked once the Room is minted; changing them requires starting a new decision / Room rather than mutating the active Room. Current 0.1.0 parameters: Search area, meal time / urgency (timestamp), dine-in vs takeout.
_Avoid_: setting, option, pre-question (ambiguous).

**Profile** (in the quiz-redesign sense):
Account-level data stored once per user and read on every decision. Identity / body / values that do not change session-to-session. 0.1.0 contents: allergies, dietary restrictions, cuisine NEVERS (negative-only). Note that the profile-edit surface itself is deferred to the pre-public-launch milestone.
_Avoid_: account info (too broad), preferences (the cuisine-craving signal is a session-level preference, distinct from profile preferences).

**Scenario question**:
A quiz question whose answer compiles to a *recipe* of multiple underlying provider request constraints at once, rather than mapping to a single filter dimension. Users pick a real-world scenario in plain language; the engine assembles the filter set. Distinct from one-question-per-filter design.
_Avoid_: composite question (mechanically accurate but doesn't convey the "scenario in user voice" framing), bundled question (collides with "bundle" in PR terminology).

**Q5 preference probe**:
The Q5 mechanism. Each member rates three real candidate venues 1-5 on excitement. The three cards preserve the strict-factorial shape -- each deviates from the member's stated Q1-Q4 profile on exactly one axis and matches the other two; never a 100% match -- but the axes are provider-aligned for Google Places: cuisine, crowd approval, and vibe. Vibe is the most expressive axis; crowd approval replaces the older Foursquare-era reputation axis. In v0.1.0 UI, Q5 cards show the place name only -- no `reviewSummary`, `generativeSummary`, or other summary text. Q5 card display order uses a deterministic app-owned shuffle per member card set; Google result order is not a display or learning signal. If a card fails current refetch before the member sees Q5, GetToIt refills first and shuffles the final three once. If a failure happens after any card is visible, GetToIt replaces only the failed slot and preserves already-seen card positions. Any replacement must preserve the failed card's original axis role (`cuisine`, `crowd_approval`, or `vibe`) so Q5 inference shape stays valid. If a visible Q5 attempt cannot preserve that shape, the member gets one automatic Q5 retry with a fresh card set under the same locked Room parameters rather than being silently dropped. Q5 retry preserves the same three axis roles and member constraints, but creates a fresh `q5_card_set_id`, fresh deterministic shuffle seed, and fresh candidate Google Place IDs. If the retry also fails, that member's Q5 remains failed; the initiator may manually close without that member's Q5 signal, or the member may exit / start a new decision depending on UI context. The engine reads the member's axis-weight hierarchy from the ratings and uses that preference signal when scoring the final group-level **Candidate pool**; v0.1.0 does not correct inference for position bias, but stores card position / display order so later scoring work can measure and correct bias from real data.
_Avoid_: archetype rater (rejected design, now historical), regret rater (the 0.1.0 mechanism, historical), tiebreaker (Q5 is a probe, not a tiebreaker -- the verdict tiebreak is the maximin rule).

**Verdict trigger** (0.1.0):
The moment a Plan's room computes a winning venue. 0.1.0 PRD locks two triggers and only two: (a) all decision-ready, non-exited participants have submitted Q5 (auto-fire on quorum-completion), or (b) the initiator manually closes voting from the Waiting surface. Manual close decides for submitted, non-exited members only; not-yet-submitted members do not impose profile constraints or preferences on that verdict. There is **no session timer** and no shot-clock countdown — 0.1.0 PRD's `timer_minutes` / `deadline_at` / cron-auto-fire mechanism is explicitly retired per 0.1.0 PRD US34, US35, and §line 115. Minimum quorum is one member (the initiator alone in the edge case where nobody else responds).
_Avoid_: auto-fire (overloaded — 0.1.0 auto-fire = "everyone finished," not "timer expired"), timer-fire (the mechanism 0.1.0 retired), shot clock (PRD voice for what was killed), Decide-now (the 0.1.0 CTA label — 0.1.0 may rename, but the underlying initiator-closes-voting action survives).

**Cuisine NEVER**:
A profile-level negative filter -- a cuisine the user will never eat under any circumstance. Hard veto, EBA-style. For group verdicts, Cuisine NEVERs combine by union: if any member has a cuisine NEVER, candidates matching that cuisine are vetoed for the whole group. Distinct from cuisine craving (session-level positive signal, "Italian sounds good tonight").
_Avoid_: cuisine dislike (too soft; NEVER is hard veto), cuisine veto (collides with the Q1-veto verdict-engine machinery from 0.1.0).

**Hard dietary safety constraint**:
A member-owned allergy or dietary restriction that must be satisfied for the recommendation to be safe enough to present. For group verdicts, hard dietary safety constraints combine by union across non-exited members: every candidate must confidently satisfy every active participant's hard allergies and dietary restrictions or be disqualified. Soft dietary preferences are out of scope for v0.1.0 unless explicitly modeled as hard constraints.
_Avoid_: dietary preference (too soft), food vibe (not a safety constraint), inferred diet fit (v0.1.0 does not infer from summaries/reviews).

**Q1 craving chip**:
A short, product-facing one-word cuisine / craving option shown in Q1. v0.1.0 chips: American, Mexican, Italian, Japanese, Chinese, Thai, Indian, Mediterranean, Middle Eastern, Korean, Vietnamese, Seafood, and Comfort Food. A chip is not a provider category; one chip may map to several Google primary types. For group verdicts, selected Q1 chips combine by union and then score by strength of group support; cuisine craving is positive signal, not a hard all-members intersection. Member count is the base cuisine-support signal, then Q5 axis-weight evidence adjusts how strongly each member's Q1 pick should matter. A No Preference answer contributes no positive cuisine signal unless every submitted, non-exited member is No Preference, in which case the final verdict fetch uses the shared no-preference union pool. Vegan is not a Q1 chip because dietary identity belongs to user-level preferences / restrictions. Breakfast is not a Q1 chip because meal timing belongs to quiz / Plan-level parameters.
_Avoid_: Google type, category chip, dietary chip.

**Q2 spend cap**:
The per-member quiz answer for the most expensive venue they are willing to consider. A cap, not an exact price bucket: choosing `$$` permits cheaper options too. Group verdicts use the strictest member cap because exceeding any member's cap breaks the hard affordability constraint. Missing provider price metadata disqualifies a venue in v0.1.0.
_Avoid_: price category, budget preference, exact spend tier.

**Vibe band**:
The per-member qualitative vibe target on a 1-5 scale. Unlike price, vibe can widen when a Q5 probe pool is too thin: a selected 4 may expand to 3-5, with the original selected value still preferred. For group verdicts, vibe targets are not intersected or enforced as hard provider filters; Q5 teaches the scoring engine each member's vibe weight, and verdict scoring uses broad room-level vibe fit where useful. Future embeddings may make the widening and fit scoring more nuanced, but the domain idea is controlled qualitative relaxation.
_Avoid_: vibe category, hard vibe filter, exact vibe match.

**Crowd approval quality floor**:
The platform-owned minimum public-trust standard GetToIt applies before showing or recommending a venue. Members do not set rating or user-rating-count parameters; those are app quality gates, unlike **Vibe band**, which is a member preference input. In v0.1.0 the floor is rating at least 3.7 and user-rating count at least 15. The floor is only a minimum; above it, stronger combined rating / count evidence remains preferred. Missing provider evidence for this floor, currently rating or user-rating count, disqualifies a venue in v0.1.0.
_Avoid_: rating display, popularity filter, reputation score, crowd slider.

**Meal-time eligibility**:
The rule that a venue should be suitable for the Plan's intended meal time. Meal time is a single shared Plan / Room **Parameter**, not a per-member quiz answer or preference to aggregate. Hard verdict eligibility, but not necessarily a native provider request filter: for Google Nearby Search, GetToIt evaluates current / regular opening-hours data server-side. Missing provider hours disqualify a venue in v0.1.0, and hours are not shown in v0.1.0 UI.
_Avoid_: open-now filter (only correct for immediate/current flows), hours display.

**Service-mode fit**:
Whether a venue supports the Plan's dine-in or takeout **Parameter**. Dine-in is a hard social-plan requirement in v0.1.0: missing or false provider dine-in evidence disqualifies a venue. Takeout is lower-risk: explicit false provider takeout evidence disqualifies, but missing takeout evidence does not.
_Avoid_: fulfillment preference, dining option, service attribute.

### Plan vocabulary (workflow overhaul, 2026-05-19)

**Plan**:
The user-facing persistent unit of intent — a named item the user creates ("Thursday dinner with the crew") and later returns to. Carries a name, a category (food only in 0.1.0), a participant scope (solo / duo / group — the *occasion* signal, not a headcount), a location, and the rest of the session parameters. A Plan is **stateful**: it moves through `pending` → `decided-active` (verdict stamped on, reroll allowed) → `decided-expired` (read-only history). Replaces the prior loose product term *decision*.
_Avoid_: decision (deprecated — too many syllables, awkward in copy), session (engineering-only term for the room+quiz run that a Plan instantiates), task (collides with Reminders-app borrow without naming the GetToIt-specific shape).

**Search area**:
A Plan / Room geographic **Parameter** set by the initiator before launch, composed of one map-selected center and radius. It is shared by the whole Room, edited together as one user-facing map control; the radius follows the visible map area and is shown with a circle plus distance label, and carries no timezone or timing semantics. Search area is a hard binary eligibility boundary: venues inside the committed circle may be considered, and venues outside it may not. Distance inside the circle does not affect scoring. Once the Room is minted, the Search area is immutable for that Room; changing it means starting a new decision / Room.
_Avoid_: location (when radius is also meant), distance (when the center is also meant), geo (engineering register).

When rendered as a Setup chip, a committed **Search area** shows the center's best human place label as the main line and the radius as supporting text (for example, "Search area · 1.5 mi"). An uncommitted value reads "Set search area."

On first open with no committed value, the **Search area editor** centers on current location with a 2.0 mi radius when permission is granted, or opens in a search-first state when permission is denied/not granted. Editing an existing Plan starts from the saved Search area. New Plans do not default to a prior Plan's area; they start from current location.

Saving a pending Plan does not require a **Search area**. Launching a Room from Setup does require a committed **Search area**; if missing, the primary action opens the **Search area editor** instead of minting the Room.

**Search area radius**:
The distance from the map camera center to the nearest visible map edge; pan changes the center, pinch zoom changes the radius, and the committed value is still persisted as center coordinate plus radius.
_Avoid_: distance slider, zoom level (implementation detail), viewport bounds (backend-inconvenient shape).

In the **Search area editor**, the radius value appears as a floating badge near the bottom center above the commit action (for example, "2.0 MI RADIUS"), outside the circle so it does not read as a map marker.

The radius badge is flanked by visible minus/plus step buttons for users who cannot comfortably pinch; the buttons use the same allowed radius stops and keep the map zoom in sync.

**Search area jump**:
A typed place search or current-location action that recenters the map before the user commits the Search area; it is navigation inside the selector, not a separate final selection mode.
_Avoid_: manual location (old C-23 language), typeahead selection (sounds final), address picker (too narrow).

**Search area editor**:
The full-screen map surface opened from the Setup chip where a member pans, pinches, jumps, and explicitly commits a Search area; leaving with uncommitted map changes prompts to commit or cancel.
_Avoid_: location sheet (old C-23 shape), map picker (underspecified), radius picker (only half the control).

The **Search area editor** chrome is intentionally minimal: close/back, a top search field, full-screen map, floating current-location button, bottom-center radius badge, and bottom primary action "USE THIS AREA." Dirty close prompts offer "Use this area" or "Discard changes."

**Density preview pin**:
A non-interactive map marker inside the selected circle for a broad Apple MapKit food/dining place shown only to help a member size the Search area; it refreshes after map movement settles, caps around 20 visible pins, and is not a candidate, recommendation, or venue detail entry point.
_Avoid_: candidate pin, result pin, restaurant pin (over-promises category/detail).

**Plan reroll window**:
The interval during which a Plan in `decided-active` state may have its verdict replaced in place. The window closes at **23:59:59 in the Plan's search-area timezone on the calendar day *after* the verdict fired** — the search-area zone is `plans.location.timeZoneIdentifier` (the IANA id the C-23 LocationPicker resolves when the coordinate is committed), NOT the creator's device timezone (locked in the sg-WF-6 grill, 2026-05-21 — the device zone is never stored, and for the common near-home Plan the two are identical anyway). E.g. a Tuesday 5 PM verdict closes Wednesday 23:59:59 (~31h); a Tuesday 11:30 PM verdict closes Wednesday 23:59:59 (~24.5h). Acceptably asymmetric — the boundary tracks the today/tomorrow mental model of the meal's location rather than a uniform clock interval. The window keeps the 0.1.0 reroll friction shape from `surfaces/07-reroll.md`: **max 3 rerolls** ("burns") per Plan, initiator-only, each with a stated app-owned reason captured as feedback / receipt metadata. In the Google **Verdict slate** model, a reroll advances to the next ranked slate entry rather than creating a new fetch constraint. A burn is consumed only when GetToIt successfully presents a viable replacement recommendation; skipped slate entries caused by current refetch or availability failure do not consume extra burns. The time window is an *outer* bound: unused burns expire when the window closes. The Plan transitions to `decided-expired` on whichever happens first — window close, third burn used, or check-in completed.
_Avoid_: reroll timer (collides with the 0.1.0-retired session timer), verdict TTL (engineering register), 24-hour window (the boundary is calendar-day-anchored, not a rolling 24h).

**Plan history**:
The list of past `decided-expired` Plans visible to the user from the Plan list surface. Read-only by definition — the verdict is preserved as a record of what was decided, not a re-runnable session. The Plan name, params, stamped verdict identity, decided date/time, winning Google Place ID, **Verdict slate**, final aggregate **Verdict fit score**, winner / reroll metadata, and app-owned receipts / cut reasons survive; full Google place display content does not. Persisted verdict rule text must be place-name-free; rule copy that names the venue is composed only after refetching current Google display data. If Google Place ID refetch fails, history shows app-owned record only: Plan name, decided date/time, "Place unavailable," and a short unavailable-details message.
_Avoid_: past decisions (uses the deprecated noun), verdict log (engineering register).

**Plan exit** (verb):
A *participant-scoped* withdrawal from an in-progress quiz, taken on any quiz screen (Q1-Q5 via an `Exit` affordance in the chrome; joiners see the same affordance labelled `Leave`). The exiter's quiz answers, Q5 votes, profile-derived constraints, and membership are removed from the active Room's decision set. The room remains alive for the remaining participants. The Plan itself does **not** change state: for the initiator's list, an exited Plan stays `pending` if the remaining members no longer hold quorum, or transitions to `decided-active` if the rest reach verdict without the exiter counted. GetToIt may keep an app-owned exit receipt, but the exiter's hard constraints no longer steer the verdict because they are no longer part of the social plan.
_Avoid_: cancel (overloaded; see Plan delete), leave (acceptable colloquially in UI copy but ambiguous about whose state changes), abandon (engineering register).

**Plan delete** (verb):
An *initiator-scoped* destruction of the Plan, taken from the Plan list surface. Kills any active room belonging to the Plan (joiners get a "session ended" toast and are punted). Removes the Plan from the initiator's list entirely. Only the Plan's creator (`creator_id` owner) can delete. Distinct from `Plan exit`: exit is "I'm out, you carry on"; delete is "everyone, we're done."
_Avoid_: cancel (overloaded), end session (joiner-side framing — joiners cannot delete), close (collides with `closed`/`hard-close` from S06).

**Plan back** (verb):
In-quiz local navigation, available on Q2 through Q5. Steps the active quiz screen one question backward (e.g., Q3 → Q2) with the prior answer preserved and re-editable. Strictly per-member — never affects others or room state. Q1 has no `Back` affordance (no prior question); the Q1 chrome carries only an `Exit` affordance instead.
_Avoid_: previous (form-field register), undo (suggests destructive reversal; back is non-destructive).

**Plan member** (any-platform):
A user participating in a Plan's quiz round. Two disjoint subtypes, distinguished by auth identity, not by role within the quiz:
- **Account member** — a user with a Linked-Apple session (mobile app holder, post-S00a). Has a personal **Plan list** surface as the app's landing. Plans they `Created` or `Joined` both appear on this list with distinct badges. May Exit any Plan, may Delete only their own Created Plans.
- **Web invitee** — a user without an account, accessing a single Plan via the iMessage/SMS deep-link rendered by the web fallback. Has **no Plan list** and no homepage; the SMS link is their only persistent handle on the Plan. Identified by a name they enter on first landing. Re-opening the same link returns them to the Plan's current state (quiz / waiting / verdict / read-only history). May Exit; cannot Delete (only creators can, and creators are always Account members).

The Plan creator is always an Account member — Plan creation is mobile-app-only post-workflow-overhaul. A Plan's joiners may be any mix of Account members (joined via mobile deep link, with the Plan appearing on their list) and Web invitees (joined via the iMessage link in a browser, no list).
_Avoid_: guest (collides with "Anonymous session"), user / participant (acceptable colloquially but underspecified about which subtype — say "Account member" or "Web invitee" when the distinction matters).

### Sheet primitives (UI dogfood, 2026-05-24)

**Modal sheet**:
A sheet hosting a rich editor surface — multi-row content, persistent open state, the user can scroll inside it. Carries the bespoke Sunset Pop sheet container: `rgba(20,20,30,0.92)` dark glass, inset 12 from edges, bottom 12, custom 38×4 handle, no native iOS grabber. Used for the reroll surface (C-16) and the C-23 LocationPicker sheet. The container is intentionally **non-native** — its visual register binds the sheet to the system's other dark-glass surfaces and is correct for modal-editor intent.
_Avoid_: bottom sheet (ambiguous — collides with `Action sheet`), reroll sheet (one specific consumer, not the primitive), dark-glass sheet (visual-property name; the *role* is what's load-bearing).

**Action sheet**:
A sheet hosting a short choice — typically 1-3 affordances and a cancel, content-height, no scrolling inside. Carries the **native mobile action-sheet shape** — rounded-top, full-width, system safe-area, native grabber on iOS. Inside the container, the Sunset Pop dark-glass register is preserved for visual continuity. Used for the S00 Plan list disambig (Solo / Group) sheet and the per-card delete-confirm sheet. Primitive: `C-2N` (assigned at the bug-24 spec edit; see [[gti-vault/15_issues/0.1.0/issues/bug-24-bottom-sheet-ios-shape|bug-24]]). Active implementation belongs in the React Native app under `mobile/`.
_Avoid_: confirm sheet (only one of two consumers is a confirm; the disambig sheet is a 2-row picker), bottom action sheet (redundant — the action-sheet idiom is bottom-anchored on iOS by construction), alert sheet (collides with iOS HIG alert, a distinct primitive).

The distinction matters because the bespoke modal-sheet container intentionally feels non-native — that is correct for the rich-editor role but **wrong** for the short-choice role, where users expect the iOS-HIG native shape. Grilling bug-24 resolved the two roles into two separate primitives instead of collapsing them.

### Verdict surfaces (pre-public-launch, 2026-05-26)

**Live verdict surface**:
The Focus surface rendered when a Plan is in `decided-active` state for an Account member who participated in the room — eyebrow + hero + time-badge + receipts + reroll + Home chrome. Carries a three-case `Flavor` enum: `default` (pre-ratify group view), `committed` (post-`I'm in` group view with dock countdown), `solo` (single-member view — suppresses receipts and time-badge audience subtitle, swaps group-save for the C-22 save-taste-profile chip). One intent: *act on this verdict before the window closes*. Reroll burns and ratify live here. See [[gti-vault/60_engineering/adr/0018-verdict-surface-three-way-split|ADR 0018]].
_Avoid_: verdict screen (overloaded — prior name for the 5-mode unified struct, deprecated 2026-05-26), Live VerdictScreen (engineering register; "live verdict surface" is the canonical noun in design-doc voice).

**Read-only verdict surface** (post-ADR-0018):
The Focus surface rendered when a viewer arrives at a `decided-expired` Plan, OR when a Web invitee deep-links into someone else's `decided-active` Plan they did not vote in. Suppresses ratify, reroll, dock countdown, save-chip; Home chrome rendering depends on arrival vector (Account member from PlanList History → yes; Web invitee from SMS → no). One intent: *show a closed verdict as a record*. Primary CTA is `Start a new decision`. Replaces the prior `.readOnly` mode case on the unified VerdictScreen.
_Avoid_: history verdict (the surface is reachable from history but also from non-history deep links — the read-only-ness is the load-bearing property, not the history-ness), past verdict screen (uses deprecated "screen" register).

**No-survivor surface** (post-ADR-0018):
The Focus surface rendered when a Plan's quiz round produces a candidate pool that the verdict engine cannot reduce to a winner. NOT a verdict surface (no hero, no time-badge, no receipts). One intent in the locked-parameter model: *end this attempt and start a new decision if the group wants different setup*. Primary CTA is `Start a new decision`; it does not widen or re-run the existing Room because Room parameters are locked after creation. Replaces the prior `.noSurvivor` mode case on the unified VerdictScreen.
_Avoid_: no-results screen (collides with the Q5 no-results screen — that one fires on per-member empty fetch, not on post-verdict no-survivor; see [[gti-vault/60_engineering/adr/0013-no-fictitious-fallback-venues|ADR 0013]]), verdict-failure screen (overloaded — "failure" implies an error path; no-survivor is a successful quiz with a recoverable outcome).

### Candidate pool (0.1.0, 2026-05-19)

**Candidate pool**:
The final group-level set of real venues a session's verdict ranks after quiz answers and Q5 preference signals are known. Under the Google provider model, Q5 may use per-member probe fetches to learn preferences, but the verdict compiles the group's constraints and learned signals into a separate server-owned verdict fetch plan. During one decision, the server may transiently dedupe the final verdict fetch against Q5 probe candidates by Google Place ID, but Q5 probe candidates are not a durable cache or guaranteed finalist list. For v0.1.0 this targets one Nearby Search request where possible; deterministic chunking is allowed if provider limits require it. Chunk results merge into one transient candidate pool by Google Place ID before eligibility cuts and scoring; chunk order, duplicate appearance across chunks, Google per-request ordering, provider ranking, and distance inside the committed Search area are not ranking or tie-breaking signals. The full scored Candidate pool is not persisted into history; durable verdict state keeps only the **Verdict slate** plus lean app-owned receipts.
_Avoid_: options (the `options` table is the pool's persisted form, not a separate concept), results (a raw provider response, pre-union), candidate set (acceptable informally, but "pool" is the canonical noun).

**Candidate-pool floor**:
The named, shared eligibility rule for venue *primary types* before any quiz answer narrows a Q5 probe fetch or the final verdict fetch. The meal-venue concept from ADR 0012 survives the Google Places switch, but the old Foursquare category-id mechanism is superseded and remapped to the Google primary-type mappings behind Q1's selectable cuisine / craving options. Q1 labels are short product-facing craving words, not provider type names; one Q1 chip may map to several Google primary types. A member's explicit Q1 picks compile to Google `includedPrimaryTypes`; "No preference" compiles to the union of all Q1-selectable category mappings, not every Google Food and Drink type. The floor does not use broad `includedTypes` such as `food`. The single source of truth that keeps Q5 probe fetches and the verdict fetch within the same meal-venue boundary.
_Avoid_: category filter (too provider-specific), restaurant filter ("Restaurant" is one member of the floor, not the whole floor).

**Verdict slate**:
The ranked app-owned result of a verdict scoring pass: up to four Google Place IDs ordered by rank, tied to the Room / verdict, with final aggregate **Verdict fit score**, winner / reroll metadata, and app-owned receipts. Ranking sorts by final aggregate score first; in v0.1.0 only exact normalized score ties use a deterministic app-owned tie-break seed derived from the Room / verdict identity and Google Place ID, unless scoring-engine tests prove a tiny versioned epsilon is needed to prevent floating-point instability. The tie-break seed is fixed for one verdict generation; rerolls do not reshuffle the slate, while a new Room / new decision gets a new seed. The first rank is shown as the initial verdict. Rerolls advance through the remaining slate without a new fetch-and-score cycle, while still refetching the selected Google Place ID for current display and basic availability. If a slate entry fails current refetch or availability, GetToIt skips to the next viable slate entry without consuming an additional reroll burn; the burn is charged only for the successfully presented replacement. If the slate is exhausted while reroll burns remain, v0.1.0 does not widen parameters, mutate the Room, or automatically start a new Google fetch. The current Room has no more recommendations to offer; the recovery CTA is `Start a new decision`. The slate never stores Google display content, raw provider payloads, provider facts such as ratings, hours, price, distance, or summaries, or component sub-scores that reconstruct those provider facts.
_Avoid_: place shortlist (suggests display content), cached recommendations (suggests provider snapshot), fallback list (the slate is the primary verdict output, not only an error fallback).

**Verdict fit score**:
The final aggregate app-owned measurement of how well a Google Place ID fit one Room's user-entered criteria and Q5 signals under a specific GetToIt scoring version. It is not an objective venue quality score and is not reusable outside that Room context. The Google provider migration locks the score's inputs, hard constraints, persistence boundary, slate / reroll rules, and tie-break invariants, but not the exact scoring formula. Component scores that directly encode Google provider facts, such as rating, price, hours, distance, atmosphere, or summary-derived scores, remain transient. In v0.1.0, the score is persisted for audit / learning but not shown in the UI.
_Avoid_: venue score (sounds objective and global), Google score (provider-owned), quality score (too broad).

**Outcome label**:
An app-owned behavioral signal created when a member or group reacts to a Q5 card or verdict, such as voted, accepted, rerolled, user-skipped, abandoned, or reason-selected. Q5 card reactions are member-scoped because they express one member's preference signal. Verdict actions are group / verdict-scoped, with `actor_member_id` when one member performs the action for the Room. System-caused skips, such as failed current refetch or slate-entry unavailability, are not outcome labels; they are operational receipts. Outcome labels may be tied to Google Place ID, Room / verdict / member context where needed, and the generator / scoring version that produced the recommendation. They do not include Google place display content, raw provider payloads, or provider facts.
_Avoid_: place label (sounds provider-owned), training data (too broad), review (collides with Google reviews).

**Reason code**:
An app-authored controlled label that explains why an app event happened, such as `wrong_vibe`, `too_expensive`, `place_unavailable`, or `selected_contrast_pool`. In v0.1.0, durable reason storage uses controlled codes only, not user free text. Reason codes are place-name-free, provider-fact-free, and stable enough for analytics; they are interpreted with their event type, Room / verdict / card context, and generator / scoring version. Released reason codes are append-only: if a meaning changes, add a new code or version instead of renaming or redefining the old code. v0.1.0 defines reason-code categories now, but exact enum values are created beside implementation: Q5 card selection, candidate keep / cut receipts, verdict outcome labels, reroll / user-skip reasons, and operational refetch / failure receipts.
_Avoid_: reason text (free prose), provider reason (not Google-authored), note (too loose).

**Google place display content**:
Provider-owned place facts and presentation fields returned by Google Places, including names, addresses, photos, ratings, hours, atmosphere fields, summaries, links, and attribution. Transient input/display content only; durable app records keep Google Place ID and app-owned verdict metadata instead. GetToIt takes the conservative policy posture: even place names are not stored durably.
_Avoid_: place snapshot, cached venue details, historical place card.

**Learning retention principle**:
When a retention choice is otherwise unresolved, GetToIt retains policy-compliant app-owned learning data in the form most useful for later scoring, analytics, and model improvement. The preferred form can vary by use case: structured outcome labels, reason codes, aggregate telemetry, versioned receipts, final aggregate fit scores, or Google Place ID references where the interaction specifically needs the Place ID. This principle never permits durable storage of Google place display content, raw provider payloads, provider facts, or provider-fact component scores.

**Provider guardrail failure**:
An app-owned failure state when the Google provider layer refuses or cannot continue because quota, per-Room call budget, or cost guardrails are exhausted. GetToIt fails closed: no hidden constraint widening, no client-controlled field-mask reduction, no attribution skipping, and no fallback provider. Durable records may keep only operational receipt data such as Room / Plan ID, failure type (`quota_exhausted`, `budget_exhausted`, `cost_guardrail_blocked`), timestamp, and step (`q5_probe`, `verdict_fetch`, `refetch`). They must not include Google payloads, display content, stale place names, or provider facts.

**Google field-mask version**:
A server-owned app contract naming which Google fields a specific use case may request. v0.1.0 uses separate stable names and versions for Q5 display, verdict display, refetch / history, and internal summary validation. Clients cannot select or widen field masks. The field-mask version may be persisted only as app-owned operational metadata when useful for debugging, cost analysis, or reproducing app behavior; it does not make returned Google fields durable.

**Google attribution payload**:
The server-provided attribution data / render instruction that accompanies any Edge Function response containing **Google place display content**. The provider layer decides when attribution is required by including this payload; UI surfaces own placement and styling, but cannot decide attribution is unnecessary. Degraded **Plan history** that shows only app-owned fallback content has no Google attribution because it displays no Google content.
_Avoid_: store everything (too broad), learning cache (suggests provider-content cache), provider archive.

**Verdict action fields**:
The minimal Google display fields shown on a v0.1.0 verdict surface so the group can act on the decision: place name, Google Maps link, and optionally formatted address. The Google Maps link is current Google display content: it is refetched or derived at render time from the current Google response by Place ID, not persisted in verdict / history. Ratings, hours, photos, summaries, atmosphere fields, and **Verdict fit score** are excluded from v0.1.0 verdict UI.
_Avoid_: place details (too broad), venue profile (suggests rich display), result card (generic).

**Google attribution**:
The Google Maps logo / text attribution and any returned third-party attribution required when GetToIt displays Google place display content outside a Google Map. In v0.1.0, even name-only Q5 cards and minimal verdict surfaces carry Google attribution because they show Google display content.
_Avoid_: credit line (too informal), legal footer (attribution belongs near the content, not buried globally).

## Relationships

- An **Anonymous session** can upgrade to a **Linked-Apple session** via `AuthCoordinator.linkApple`, preserving the `user_id` and all owned rows (rooms, votes, members, events).
- The **S00a Sign-in Gate** is the mobile-app entry point that converts any non-Linked-Apple state into a **Linked-Apple session**. It is the ONLY surface that mints a Linked-Apple session in the app.
- A **Linked-Apple session** is durable across reinstalls keyed by the device's Apple identity. An **Anonymous session** is durable only until the keychain is cleared or the app is deleted.
- An **Account claim** reuses the existing S00a `linkApple` path unchanged — its only new work is transporting the Anonymous session from browser to app keychain via a **Claim code**. Once the session is in the keychain, S00a treats the user identically to a pre-S00a legacy-carryover anonymous install.
- A **Parameter** is set on session setup and constrains the entire quiz that follows. A **Scenario question** is asked of every member during the quiz, and its answer is per-member. A **Profile** value is read silently per member without re-asking.
- The conditional Q3 structure (indoor/outdoor only if dine-in **Parameter**) means quiz question identity depends on session parameters. Schema must support per-session question-slot variability.
- The active Room decision set contains only non-exited members. If a member exits before verdict, their Q1-Q5 answers, Q5 card ratings, price cap, Cuisine NEVERs, hard dietary safety constraints, and vibe signals stop contributing to the final **Candidate pool** and verdict score. Exit receipts are app-owned audit state, not continuing preference input.
- If the initiator manually closes voting before every non-exited member has submitted Q5, the verdict decision set contains only submitted, non-exited members. Not-yet-submitted members do not impose profile constraints, hard dietary safety constraints, Cuisine NEVERs, price caps, cuisine choices, or vibe signals on the final **Candidate pool**. The Waiting surface should make clear that closing now decides for the submitted group only.
- The **Q5 preference probe** uses per-member probe fetches to learn preference weights; the verdict engine then builds a final group-level **Candidate pool** from the group's constraints and Q5 signals. They are separate fetch stages, but both obey the same **Candidate-pool floor** and Google storage boundary.
- Q5 probe candidates may be reused only transiently inside the same server-side decision for in-memory dedupe or scoring context. The final verdict still runs its own group-level fetch / scoring cycle. Q5 cards are preference probes first, not a hidden finalist cache, and they must not be promoted into durable provider-content storage.
- If provider limits force deterministic chunking for the final verdict fetch, GetToIt merges all chunk results into one transient set by Google Place ID, applies the same eligibility cuts once, then scores from app-owned rules and transient provider facts. A place that appears in multiple chunks is still one candidate with one score; duplicate appearance is a provider artifact and must not create a boost. Chunk order and Google per-request ordering must not influence ranking.
- In v0.1.0, every candidate that survives the merged final **Candidate pool** eligibility cuts is scored. The provider fetch / chunk budget should keep that merged pool bounded. If future scale requires pre-score downselection, it needs a separate explicit design; provider order, distance inside the Search area, or duplicate chunk appearance must not be used as shortcuts.
- Q5 card / vote telemetry may include Google Place ID as app-owned interaction state tied to room / member / card / vote records, so Q5 inference and verdict ranking know which candidate was rated. Q5 card display order uses a deterministic app-owned shuffle keyed to the member and `q5_card_set_id`: reopening the same set shows the same order, while a new card set gets a new seed. Google result order must not determine card order. If a Q5 card fails current refetch before the member sees the Q5 set, GetToIt refills the set first and shuffles the final three cards once. If same-axis refill cannot be found before display, GetToIt regenerates the Q5 set or takes the Q5 no-results path. If a failure happens after any card is visible, GetToIt replaces only the failed slot with a same-axis card, preserves already-seen card positions, and records an app-owned operational receipt; if no same-axis replacement is possible, that member's Q5 attempt fails rather than silently changing inference shape. A failed visible Q5 attempt permits one automatic retry with a fresh card set under the same locked Room parameters. Retry preserves the same three axis roles and member constraints, but uses a fresh `q5_card_set_id`, fresh deterministic shuffle seed, and fresh candidate Google Place IDs. If the retry also fails, that member's Q5 remains failed; they are not silently dropped, but the initiator may manually close without their Q5 signal under the existing submitted-member rules, or the member may exit / start a new decision depending on UI context. v0.1.0 inference does not correct for card position bias; it persists position / display order so future scoring work can measure bias and add correction with real data. After verdict / history, retain only the member-scoped Q5 learning telemetry, not the full rendered Q5 card set. Durable Q5 telemetry may keep Place ID, card position / display order, card axis role, rating / outcome label, `q5_card_set_id`, `q5_generator_version`, `constraint_profile_version`, and app-authored, place-name-free selection reason codes such as `selected_cuisine_keep_feasible`, `selected_contrast_pool`, `selected_vibe_band_edge`, or `selected_crowd_quality_floor`. It must not include **Google place display content**, a rendered-card snapshot, raw candidate payloads, provider facts inside reason text, or provider-derived candidate facts beyond the Place ID.
- If a shown Q5 card receives no rating because the member exits or leaves the quiz incomplete, retain a member-scoped non-response **Outcome label** rather than a rendered-card snapshot. The durable record may keep Place ID, card position, axis role, generator / constraint version, and controlled reason code such as an exit or incomplete reason. It must not include **Google place display content** or raw candidate payloads.
- Candidate eligibility receipts may persist Google Place ID plus app-owned keep / cut reason codes when tied to an active Room, Q5 card set, or verdict decision. They must stay lean: no provider values behind the reasons, no place names, no addresses, no ratings, no hours, no price, no distance, no summaries, and no raw payloads. Broad candidate-pool receipts are active-Room audit state, not a durable place archive; only app-owned verdict receipts survive into **Plan history**.
- The full scored verdict **Candidate pool** is transient and must not be persisted as a historical archive of every scored Place ID, rank, and score. Durable verdict state keeps the top-four **Verdict slate** plus lean app-owned receipts needed for audit / learning.
- A **Verdict slate** stores the top four ranked Google Place IDs from the verdict scoring pass, with final aggregate **Verdict fit score** and scoring version, so rerolls can advance to rank 2, 3, or 4 without another fetch-and-score cycle. Ranking sorts by final aggregate score first. In v0.1.0, only exact normalized score ties are broken by a deterministic app-owned seed derived from the Room / verdict ID and Google Place ID, unless scoring-engine tests prove a tiny versioned epsilon is needed to prevent floating-point instability. The tie-break seed is fixed for the verdict generation: rerolls advance through the existing order and do not reshuffle tied entries; a new Room / new decision gets a new seed. The tie-breaker must not use Google result order, chunk order, duplicate appearance, provider ranking, or distance inside the committed Search area. It must not persist provider-fact component scores such as rating score, price score, hours score, distance score, atmosphere score, or summary-derived score. Before showing any slate rank, GetToIt refetches current display data and basic availability by Google Place ID; unavailable ranks are skipped or handled through the existing reroll / refetch-failure rules. Provider/availability skips do not consume reroll burns; one burn is consumed only when a viable replacement is presented. If the slate is exhausted before the Plan uses all reroll burns, v0.1.0 does not auto-fetch a new slate or change Room parameters; the current Room has no more recommendations to offer, and the user starts a new decision if they want different parameters.
- The Google provider migration locks scoring inputs and invariants, not the final scoring formula. Hard constraints, provider-field eligibility rules, persistence boundaries, **Verdict slate** / reroll behavior, and tie-breaking rules are part of this decision. Exact weights, normalization, embeddings behavior, and formula tests belong to the future scoring-engine / embeddings design work.
- **Outcome labels** are durable app-owned learning signals and may reference Google Place ID because they record how users reacted to GetToIt's own card or verdict. Q5 votes and card reactions are member-scoped. Verdict acceptance, reroll, user-skip, abandoned, and reason-selected events are group / verdict-scoped, with `actor_member_id` when one member takes the action. Store them with the relevant Room / verdict / member context, generator / scoring version, and app-owned reason codes. Do not store Google names, provider facts, summaries, or raw candidate payloads beside them; future analysis can refetch current Google data by Place ID when policy allows and the analysis needs it.
- The **Learning retention principle** applies to future data-retention choices: retain useful app-owned learning signals when policy-compliant, choosing the structure that best fits the use case, but do not cross the Google storage boundary.
- v0.1.0 durable reason storage uses controlled **Reason codes** only. Store each code with event type and generator / scoring version. Released reason codes are append-only; if meaning changes, add a new code or version rather than renaming or redefining the old code. The durable categories are Q5 card selection, candidate keep / cut receipts, verdict outcome labels, reroll / user-skip reasons, and operational refetch / failure receipts. Exact enum values are deferred until implementation so they match real event shapes. User-authored free-text reasons are out of scope because they are harder to analyze, carry privacy risk, and could accidentally persist Google display content or provider facts.
- A selected **Q1 craving chip** narrows the member's Google fetch but does not seal the member inside that cuisine. The fetch planner also adds a contrast / no-preference pool from the union of unselected Q1 mappings so Q5 can still generate a cuisine-drop card. A "No preference" member uses only the no-preference union pool.
- For the final group-level **Candidate pool**, selected **Q1 craving chips** combine by union for the verdict fetch plan, with contrast / no-preference coverage included only where it stays inside the server-owned call budget and provider request limits. No Preference members add no positive cuisine pressure when other submitted, non-exited members selected cuisines; they are represented by the absence of cuisine pressure. If every submitted, non-exited member is No Preference, the final verdict fetch uses the shared no-preference union pool. Verdict scoring should prefer cuisines with stronger member-count support, but Q5 can adjust that support by showing whether each member actually weights cuisine highly. The exact scoring formula remains deferred to the scoring-engine / embeddings design work.
- For group verdicts, **Cuisine NEVERs** combine by union as hard vetoes. This is intentionally the opposite of **Q1 craving chips**: positive cuisine cravings are soft / union / scored, while negative Cuisine NEVERs are hard / union / vetoed.
- When a member selects exactly one **Q1 craving chip**, two of their three **Q5 preference probe** cards should come from that selected chip's mapping and one should come from the contrast pool.
- When a member selects two or three **Q1 craving chips**, the **Q5 preference probe** still shows two cuisine-axis keep cards and one cuisine-drop / contrast card; the keep-card cuisines are chosen by member-local card-generation feasibility, and any unshown selected chip still contributes to scoring / verdict.
- A **Search area** is a single initiator-set Room parameter. It supplies only the shared geo/radius eligibility boundary for Q5 probe fetches and the final **Candidate pool** fetch; it does not decide Plan timing or timezone behavior, and members do not contribute separate locations / radii in v0.1.0. Distance is binary for eligibility and must not be used as a ranking / scoring signal inside the committed circle.
- The Google provider layer may overfetch slightly beyond the committed **Search area** to offset provider ranking / edge effects, using committed radius plus `min(radius * 0.15, 0.5 mi / 805 m)`. This overfetch radius is not user-visible. **Candidate pool** eligibility is still trimmed server-side to the committed **Search area** before Q5 card generation or verdict ranking.
- Overfetch telemetry may record only non-place aggregates such as committed radius, provider request radius, overfetch delta, candidate counts before / after trim, and aggregate trim reason counts. It must not record **Google place display content**, raw provider responses, Google Place IDs, or other per-place identifiers.
- Operational logs should prefer room / candidate IDs, aggregate counts, status / error codes, latency, field-mask / SKU labels, and reason-code summaries over Google Place IDs. Google Place IDs may appear only in structured, access-controlled server logs when needed for debugging, with short retention. Analytics and event streams should not carry Google Place IDs unless they are app-owned interaction records that explicitly need the Place ID, such as Q5 votes or verdict receipts.
- The Google provider layer owns a per-Room call budget for candidate fetches and refetches. Fetch planning is deterministic, retries are bounded, and clients cannot widen field masks or increase call counts. For v0.1.0, the fetch / chunk budget should bound the merged final pool so every eligible candidate can be scored. If the budget cannot produce enough valid candidates after allowed qualitative relaxation, GetToIt takes the no-results path rather than silently expanding cost or constraints.
- If Google quota, the per-Room call budget, or cost guardrails are hit mid-Room, GetToIt fails closed into no-results / temporary-unavailable handling. It must not widen constraints, reduce field masks client-side, skip attribution, or fall back to another provider. Durable telemetry is an app-owned **Provider guardrail failure** receipt only: Room / Plan ID, failure type (`quota_exhausted`, `budget_exhausted`, `cost_guardrail_blocked`), timestamp, and step (`q5_probe`, `verdict_fetch`, `refetch`), with no Google payload, display content, or provider facts.
- Google field masks are server-owned named / versioned contracts per use case: Q5 display, verdict display, refetch / history, and internal summary validation. Clients cannot choose or widen them. Durable operational metadata may include **Google field-mask version** for debugging, cost analysis, or behavior reproduction, but must not persist returned Google fields because that version exists.
- A **Search area jump** can change the center of a **Search area**, but the committed **Search area** remains the map's current center plus **Search area radius**.
- The **Search area editor** is opened from one compact Setup chip; the map does not render inline on Setup.
- **Density preview pins** can appear inside the **Search area editor**, but they do not define or promise membership in the **Candidate pool**.
- The Google Places migration reopens the older "fetch vs score" split for Q1-Q4. Session requirements that users experience as hard constraints -- Search area, meal/open timing, price cap, dine-in/takeout fit, dietary hard needs, and **Cuisine NEVERs** -- may become fetch-time Google request constraints. Positive cuisine craving and vibe preference remain softer scoring / probe signals unless a Google filter is broad enough that it does not collapse the **Candidate pool**.
- v0.1.0 has no production users, so the Google provider migration removes the Foursquare / MapKit provider implementation entirely rather than keeping dormant fallback paths. Google Places is the only candidate-pool provider; any old FSQ / MapKit runtime path, config, provider key, cache-table dependency, active schema artifact, environment variable, function, migration path, telemetry stream, or fallback branch is migration debt to delete, not compatibility surface to preserve. Active DB / code should speak Google Place ID and app-owned records, not `fsq_*`. Active Supabase migration history should be rewritten / squashed into a clean baseline so a fresh DB reset does not recreate FSQ / MapKit tables, columns, functions, or provider config; historical rationale belongs in ADRs, not executable migrations.
- **Q2 spend cap** and **Meal-time eligibility** are hard candidate eligibility rules even when Google Nearby Search does not support them as native request filters. For group verdicts, the effective **Q2 spend cap** is the strictest member cap, while **Meal-time eligibility** comes from the single shared Plan / Room meal-time **Parameter**. The Edge Function may evaluate these rules from returned current Google fields; missing price or hours metadata disqualifies a candidate in v0.1.0.
- **Service-mode fit** is asymmetric. Dine-in Plans require explicit dine-in support; missing or false provider evidence disqualifies. Takeout Plans disqualify only on explicit false takeout evidence; missing takeout evidence does not disqualify.
- **Hard dietary safety constraints** and **Cuisine NEVERs** are not inferred from Google summaries, reviews, atmosphere text, or future summary embeddings in v0.1.0. They combine by union across non-exited group members and are enforced only from user-owned constraints and high-confidence provider fields / types. If a hard allergy or dietary restriction cannot be confidently enforced for a candidate, that candidate is disqualified; soft dietary preferences are out of scope unless explicitly modeled as hard constraints.
- Group **Vibe band** targets are scoring inputs, not hard verdict-fetch intersections. If the Google provider request needs a vibe-related shape, it should use a broad room-derived band inside the server-owned call budget; the final recommendation is chosen by scoring Q5-inferred vibe importance against the final **Candidate pool**.
- When a Q5 probe pool is too thin, GetToIt relaxes qualitative constraints before hard constraints. **Vibe band** widens first, preserving preference for the selected value. The **Crowd approval quality floor** is platform-owned rather than member-selected; any relaxation of crowd approval can only happen inside that v0.1.0 floor. Price cap, meal-time eligibility, required provider metadata, and selected Q1 cuisine mappings stay fixed for the current Room. Changing them requires starting a new decision / Room. Cuisine still follows the already-accepted keep / contrast design rather than a sealed-box filter.
- The **Crowd approval quality floor** applies before Q5 card generation and again before final verdict ranking as a shared app standard, not a strictest-member or averaged group preference. Final preference still rewards stronger combined rating / count evidence above the floor. The exact scoring formula is intentionally unresolved until the scoring-engine / embeddings design work.
- Missing Google atmosphere fields or summaries hurt vibe scoring confidence but do not disqualify a candidate in v0.1.0. Missing crowd approval evidence disqualifies a candidate because Q5 and verdict trust require enough public signal to recommend it.
- **Google place display content** may appear on Q5, live verdict, and read-only verdict surfaces only after a current fetch by Google Place ID or current candidate-pool fetch. It is not durable **Plan history**.
- Google Maps URI / link follows the same storage boundary as place name and address: it may be returned in the current Edge Function response for action display, but is not persisted in verdict / history. Durable state keeps Google Place ID, and the UI receives the current link only after refetch / derivation by Place ID.
- **Google attribution** appears whenever **Google place display content** appears. Any Edge Function response containing Google display content must include a **Google attribution payload** sufficient for the client surface. UI surfaces own placement and styling, but cannot suppress attribution or decide it is unnecessary. Degraded **Plan history** that shows only app-owned fallback content does not show Google attribution.
- Google refetch failures are context-sensitive. Server-side refetch may use bounded retries before taking the final app action: one immediate retry plus one short backoff for transient 429 / 5xx / network-timeout failures. Do not retry hard failures such as `NOT_FOUND`, permission or key errors, invalid field masks, or policy / config errors. Clients do not run Google retry loops; they receive the final app decision. Q5 drops the unavailable candidate and refills from the current Q5 probe pool; if it cannot form three valid cards, it takes the Q5 no-results path. A pre-display Q5 refill preserves the failed card's axis role and shuffles the final three-card set once; if same-axis refill cannot be found, GetToIt regenerates the set or takes the Q5 no-results path. A post-visible Q5 refill replaces only the failed slot with a same-axis card and preserves already-seen positions; if no same-axis replacement is possible, that member's Q5 attempt fails and the member may retry once with a fresh Q5 card set under the same locked Room parameters. Q5 retry preserves the original axis recipe and member constraints, but does not replay stale candidates: it gets a fresh `q5_card_set_id`, fresh shuffle seed, and fresh candidate Google Place IDs. If the retry also fails, that member's Q5 remains failed and the app stops retrying. A `decided-active` **Live verdict surface** whose winning Google Place ID cannot refetch forces reroll / recompute. Only history / read-only contexts show degraded app-owned verdict content. Refetch failures may be durably recorded only as app-owned operational failure events: Google Place ID, room / Plan / verdict ID, failure type, timestamp, and action taken, such as `drop_card`, `replace_card_slot`, `q5_attempt_failed`, `q5_retry_started`, `q5_retry_exhausted`, `reroll_required`, `slate_entry_unavailable`, or `history_degraded`. They are not **Outcome labels** because they describe technical availability rather than user / group reaction. They must not store failed response bodies, stale display content, or last-known place names, and they must not become a durable negative cache.
- A **Plan** is the durable owner; a **Room** is the ephemeral container for one quiz round. A Plan owns at most one active Room (the in-flight round). Launching a `pending` Plan mints a new Room. On verdict, the Room closes and the Plan transitions to `decided-active`. A reroll re-runs the verdict on that **same** Room in place — it does not mint a new Room (the 3-burn cap is per-Room). Once the Plan transitions to `decided-expired`, no new Rooms may be minted against it. Plan name, status, params, and the stamped verdict survive Room lifecycle; the Room's `votes` / `options` rows do not.
- Room setup is immutable after launch. Search area, meal timing, dine-in / takeout mode, price caps, cuisine answers, dietary hard needs, and other quiz/setup constraints are not edited inside an active or decided Room. Changing them means starting a new decision / Room, not widening the existing one.
- A **Modal sheet** and an **Action sheet** are distinct primitives, not visual variants. They differ in *role* (rich editor vs short choice), in container shape (bespoke dark-glass vs native platform action-sheet), and in platform semantics. C-16 (reroll) and C-23 LocationPicker are Modal sheets. The S00 disambig (Solo / Group) and delete-confirm sheets are Action sheets. A future surface that needs to host a richer single-choice picker (e.g. a long list) belongs in the Modal sheet family, not the Action sheet family — the choice is editor-vs-choice, not list-length.

## Example dialogue

> **Founder:** "There is still no sign-up page upon app launch."
> **Engineer:** "The **S00a Sign-in Gate** is wired but its render guard only fires when there is no session at all. Your phone still has an **Anonymous session** in keychain from a pre-S00a build, so the gate is being skipped."
> **Founder:** "And when I do sign in, I keep my old rooms?"
> **Engineer:** "Yes. The fix routes you through **S00a** but taps `linkApple` instead of `signInWithApple`, so your **Anonymous session** is upgraded to a **Linked-Apple session** with the same `user_id`. Every row keyed off that id survives."

## Flagged ambiguities

- "sign-up" vs "sign-in": product / founder voice uses "sign-up page" loosely for any account-creation moment. Engineering uses "sign-in gate" because the surface only ever signs in with an existing Apple identity (Apple owns account creation; the app never sees a "new vs returning" distinction). When the founder says "sign-up page," they almost always mean the **S00a Sign-in Gate**.
- "anonymous user" vs "no user": pre-S00a these were the same in practice (every legacy mobile install minted an anonymous user on launch). Post-S00a they diverge - no user = pre-S00a state; anonymous user = legacy carryover that must pass through S00a via `linkApple`.
- "Search-area timezone" conflict: the existing **Plan reroll window** definition says the close boundary uses `plans.location.timeZoneIdentifier`, but the Search area grill on 2026-06-03 clarified that **Search area** has no timing or timezone semantics. The timing model needs a separate correction; do not use the map selector work to reinforce the old search-area-timezone language.
